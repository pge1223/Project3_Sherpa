# AI Review Board

**문서를 평가하는 AI가 아니라, 문서를 놓고 전문가들이 회의하는 AI 위원회**

---

## 01. 어떤 아이디어인가

공모전 기획서, 창업 사업계획서, 정부 지원사업 제안서, 투자 IR 덱, 졸업 작품, UX 기획서, 논문 제안서 — 형태는 다르지만 모두 "검토받아야 하는 문서"라는 공통점을 가진다. 기존 AI 첨삭 도구는 문서를 넣으면 피드백 하나가 나오는 단선적인 구조였다.

AI Review Board는 여기서 한 단계 더 들어간다. 문서를 업로드하면 관련 공고문·평가기준·가이드라인을 RAG로 수집하고, 그 기준에 맞는 **전문가 페르소나(위원)를 자동으로 구성**한다. 각 위원이 독립적으로 문서를 검토하고, 위원장이 이를 종합해 실행 가능한 피드백을 돌려준다. 결과물은 평가 점수가 아니라, 실제 회사 회의를 지켜보는 듯한 경험이다.

---

## 02. 어떻게 쓰이는가 — 회의 진행 순서

| 의안 | 단계 | 설명 |
|---|---|---|
| 1 | 문서 업로드 | 사업계획서, 공모전 기획서, IR 덱 등 검토받을 문서를 올린다. |
| 2 | 공고문 · 평가기준 RAG 수집 | 해당 공모전/사업의 공고문, 심사 기준, 가이드라인을 검색해 문서의 성격을 파악한다. |
| 3 | 위원회 구성 | 문서 종류에 따라 필요한 역량을 추출해 위원 페르소나를 매칭한다. 회의 참석자와 회의 문화가 도메인마다 달라진다. |
| 4 | 1차 회의 — 개별 리뷰 | 각 위원이 서로의 발언을 보지 않고 독립적으로 문서를 검토해 평점과 근거를 남긴다. |
| 5 | 2차 회의 — 위원장 종합 | 위원장 역할의 AI가 1차 의견 전체를 입력받아 쟁점을 정리하고, 액션 아이템을 제시한다. |
| 6 | 회의록 반환 · 반복 검토 | 사용자는 회의록을 바탕으로 문서를 수정하고, 다시 회의를 열어 개선 여부를 확인할 수 있다. |

---

## 03. 위원회 구성 예시 — 문서마다 참석자가 바뀐다

의안 3에서 구성되는 위원회는 고정된 한 팀이 아니다. 문서가 어떤 성격인지에 따라 **참석하는 페르소나 자체가 달라지고**, 각자 자기 전공 분야의 언어로 코멘트를 남긴다.

**공모전 · 일반 사업계획서**
- PM — "문제 정의는 좋은데, 이 기능은 MVP에서 빼도 될 것 같습니다."
- 개발자 — "이 부분은 구현 난이도가 꽤 높습니다."
- 마케터 — "타겟층이 조금 넓게 잡힌 것 같아요."
- 사업팀 — "수익 모델(BM)이 아직 약합니다."

**정부 지원사업 제안서**
- 공공성 전문가 — "사회적 가치 측면의 설명이 부족합니다."
- 정책 전문가 — "관련 정부 정책과의 연결고리가 필요합니다."
- 예산 전문가 — "예산 산정 근거를 좀 더 보완해야 합니다."

**디자인 · UX 공모전**
- UX 전문가 — "핵심 사용 흐름이 다소 복잡합니다."
- 디자이너 — "기존 서비스 대비 차별성이 부족해 보입니다."
- 접근성 전문가 — "접근성 가이드라인 고려가 필요합니다."

각 위원의 코멘트는 감(感)이 아니라, RAG로 수집한 공고문·평가기준·근거 자료를 바탕으로 나온다. 위원장은 이 코멘트들을 모아 최종 회의록으로 종합한다.

---

## 04. 핵심 차별점

**Point 1. RAG는 답이 아니라 위원을 만드는 재료**
일반적인 RAG는 정보를 검색해 답변에 쓰지만, 여기서는 공고문·평가기준을 분석해 그에 맞는 전문가 페르소나를 생성하는 데 쓴다.

**Point 2. 평가가 아니라 회의**
토론이 아닌 회의 형식으로, 각 위원이 자기 관점과 근거를 들어 발언하고 위원장이 종합한다. 실제 회사 의사결정 과정과 닮아있어 몰입도가 높다.

**Point 3. 도메인마다 달라지는 회의 문화**
스타트업, 정부지원사업, 디자인 공모전마다 위원 구성(템플릿)과 논의 순서는 정해두되, 공고문 분석 결과에 따라 위원별 평가 기준(rubric)이 동적으로 조정된다. 안정성과 맞춤성을 함께 확보하는 구조다.

## 05. 기술 스택

웹 기반 서비스로 구현하며, 이 프로젝트도 하나의 위원회처럼 각 기술이 맡은 역할로 참여한다.

| 영역 | 기술 | 역할 |
|---|---|---|
| Frontend | React + Vite | 회의실 UI, 위원별 발언 화면 구성 |
| Backend | FastAPI | RAG·에이전트 파이프라인과 프론트엔드를 연결하는 API 서버 |
| RAG | LangChain · KURE-v1 · Chroma (Persistent) | 공고문·평가기준 수집 및 임베딩 검색, 위원 페르소나 생성의 기반 자료 확보 |
| Workflow | LangGraph | 위원별 독립 리뷰(1차) → 위원장 종합(2차)으로 이어지는 전문가 회의 오케스트레이션 |
| LLM | OpenAI / Anthropic API | 역할별로 경량·고성능 모델을 분리 사용해 비용과 품질의 균형을 맞춤 |
| Document Parsing | PyMuPDF · python-docx · python-pptx | PDF/워드/PPT 형태로 업로드되는 사업계획서·기획서 파싱 |
| Database | MongoDB | 사용자 문서, 회의 기록, 위원 평점 데이터 저장 |
| Infra / Hosting | NCP (Naver Cloud Platform) | 웹 기반 서비스로 서버 호스팅 및 배포 |
| Human AI | TTS + LivePortrait / Wav2Lip (Colab GPU) | 위원들이 실제로 발언하는 회의실 시각화 — 텍스트가 아닌 "회의를 보는" 경험 구현 |
| Monitoring | LangSmith / 자체 로그 | 프롬프트·검색 품질 추적, 근거 인용 정확도 모니터링 |
