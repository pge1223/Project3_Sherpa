# MuseTalk 스트리밍 지연 최적화 작업 기록

`/generate-stream`(WebSocket)에서 첫 청크(첫 화면)가 뜨기까지 걸리는 시간을 줄이기 위해
시도한 것들과 결과를 기록합니다. 목표는 "대화 시작하면 2~3초 안에 응답 영상이 뜨는 것"이고,
이 문서는 그 목표에 도달하기까지의 진단 과정과 근거를 남깁니다.

관련 파일: `ai/media/musetalk/colab/musetalk_setup.ipynb`(6-1번 섹션 = MuseTalk 소스 패치,
11번 섹션 = FastAPI `app.py`), `ai/media/musetalk/web_test_stream.html`(스트리밍 테스트 페이지).

---

## 환경

- GPU: Google Colab A100
- MuseTalk: v1.5 (TMElyralab/MuseTalk 공식 레포)
- 추론: PyTorch (TensorRT 미사용 — 아래 "TensorRT를 시도하지 않은 이유" 참고)
- TTS: OpenAI `gpt-4o-mini-tts`
- 페르소나 A 원본 루프 영상: `data/persona_a/avata_rf.mp4` (241프레임)

## 초기 상태

FastAPI 서버(`app.py`)는 계속 떠 있지만, 실제 립싱크 작업은 요청마다
`asyncio.create_subprocess_shell("python -m scripts.inference ...")`로 **완전히 새로운
프로세스**를 띄워서 처리하고 있었다. 최초 측정 결과:

- 첫 청크까지: 약 74초
- `[TIMING]` 로그로 확인한 결과, 그중 **69.27초가 "PNG 프레임 파일이 디스크에 하나도 없는" 시간**

## 시도 1 — 배치 인터리브 (실패, 원인 오진단)

**가설**: MuseTalk 공식 `scripts/inference.py`가 "오디오 전체에 대해 UNet+VAE 추론을
전부 끝낸 뒤 → 그다음에 프레임 블렌딩+저장을 시작"하는 완전히 분리된 2단계 구조라서,
마지막 배치까지 다 끝나야 첫 PNG가 생긴다고 추정.

**조치**: 두 루프를 배치 단위로 합쳐서, 배치 하나가 추론되면 즉시 그 배치를
블렌딩+저장하도록 `scripts/inference.py`를 패치.

**결과**: 69.27s → 66.72s. **거의 개선되지 않음.**

**재진단**: `tqdm` 로그를 보면 배치 추론 루프 자체는 "MuseTalk 시작 후 63초"가 지나야
겨우 진행되기 시작했다. 즉 **병목은 배치 루프 구조가 아니라, 배치 루프에 진입하기도 전인
`main()` 앞부분**에 있었다. 이 패치 자체는 (해는 없고 구조적으로 맞는 방향이라) 유지하되,
진짜 원인을 더 찾기 위해 세분화된 타이밍 로그(`[INF-TIMING]`)를 추가.

## 시도 4 — 지연(lazy) import (성공, 실측 완료)

캐시 히트 상황에서도 `main() 시작`과 실제 배치 루프 사이에 예상보다 큰 시간차가 있었다.
원인: `scripts/inference.py`가 파일 최상단에서 `musetalk.utils.preprocessing`을 무조건
import하는데, 이 모듈은 **import되는 순간 모듈 최상단 코드가 실행되며 DWPose 포즈 모델 +
얼굴 감지 모델을 GPU에 자동으로 로드**한다:

```python
# musetalk/utils/preprocessing.py 최상단
model = init_model(config_file, checkpoint_file, device=device)  # DWPose
fa = FaceAlignment(LandmarksType._2D, flip_input=False, device=device)  # 얼굴 감지
```

캐시 히트 시에는 랜드마크를 재계산할 일이 없어 이 두 모델을 아예 쓰지 않는데도, import
시점에 무조건 GPU에 올라가고 있었다. 이 import를 **캐시 미스일 때만 실행되는 지연
import**로 옮김 (`else:` 블록 안으로 이동).

**실측 결과**: 첫 청크까지 40.5s → **약 33~35s**.

## 남은 병목 — 모델 로드 (subprocess-per-request 구조 문제)

시도 3·4 이후 cache-hit 요청의 시간 구성:

| 구간 | 시간 |
|---|---|
| TTS(OpenAI API) | 2.5~4.6s |
| **VAE/UNet/PE 모델 로드** | **~12s** ← 압도적 1위 |
| Whisper+FaceParsing 초기화 | ~0.3s |
| 캐시 로드(241프레임) | ~4.2s |
| 오디오 특징 추출 | ~2.5s |
| 배치 루프(첫 프레임까지) | ~8s |

**왜 캐싱으로 못 줄이나**: `app.py`가 요청마다 `python -m scripts.inference`를 **새
프로세스**로 띄우기 때문에, 그 프로세스가 종료되면 GPU에 올라간 모델도 함께 사라진다.
다음 요청은 또 처음부터 CUDA 초기화 + 모델을 디스크에서 GPU로 로드해야 한다. 이건
GPU를 더 빠른 걸로 바꿔도(A100이라도) 사라지지 않는 **구조적 비용**이다 — "얼마나
빠르게 로드하느냐"가 아니라 "매번 로드해야 하느냐"의 문제이기 때문.

같은 이유로, 2차 프로젝트(RTX 4060, `model_test.md` 참고)에서 썼던 GPU 워밍업 기법도
지금 구조에서는 무의미하다. 워밍업 캐시는 그 프로세스가 살아있는 동안만 유효한데,
지금은 매 요청마다 프로세스가 죽었다 새로 뜨기 때문이다.

## 시도 5 — 모델 상주 구조로 `app.py` 리팩터링 (성공, 실측 완료)

**방향**: `python -m scripts.inference`를 subprocess로 띄우는 대신, FastAPI 서버
시작 시(`@app.on_event("startup")`) UNet/VAE/PE/Whisper/FaceParsing을 **한 번만**
GPU에 로드해서 전역 상태로 들고 있고, 요청은 이미 로드된 모델을 함수로 직접 호출한다.
페르소나 준비 캐시도 디스크 파일뿐 아니라 **프로세스 메모리**에도 유지해서, 서버가
살아있는 한 페르소나 준비를 다시 할 일이 없게 한다.

- `load_models()`: 서버 시작 시 1회 호출, 모델을 전역 변수에 로드
- `_prepare_persona(video_path)`: 메모리 캐시 → 디스크 캐시 → (둘 다 없으면) 최초 계산,
  3단 폴백. `PERSONA_MAP`에 있는 페르소나는 서버 시작 시 미리 워밍업.
- `run_inference_sync(...)`: `scripts/inference.py`의 배치 추론 루프를 그대로 가져와
  함수화. 블로킹(GPU) 함수라 WebSocket 핸들러에서는 `ThreadPoolExecutor`로 스레드풀에서
  실행해 이벤트 루프를 막지 않음 (동기 `def` 라우트인 `/generate`는 FastAPI/Starlette가
  자동으로 스레드풀에서 실행해주므로 별도 처리 불필요).
- 프레임을 폴링해서 청크 단위로 인코딩+전송하는 기존 WebSocket 로직(파일 diff 감지,
  1초 분량씩 묶어 ffmpeg 인코딩 후 전송)은 그대로 유지 — 변경 범위를 "모델을 어떻게
  띄우느냐"로 최소화해서 이미 검증된 스트리밍 로직의 리스크를 줄임.

**실측 결과** (서버 시작 시 모델 로드 12.83s + 페르소나 워밍업 4.18s, 이후 모든 요청이 재사용):

| | 서버 기동 직후 첫 요청 | 완전히 워밍업된 2번째 요청 |
|---|---|---|
| 첫 프레임 파일 발견 | 7.82s | **3.22s** |
| **첫 청크(재생 가능한 영상) 전송** | 11.27s | **6.64s** |

**첫 청크까지 6.64초** — 초기 상태(~74~78s) 대비 **약 11~12배 단축**. 이 중 TTS(OpenAI API 호출)가 2.7초를 차지하므로, TTS 완료 이후 실제 생성 파이프라인 자체는 약 3.8초 만에 첫 화면을 만들어낸다. 최초 목표("대화 시작 2~3초 안에 응답")에 근접.

서버 기동 직후 첫 요청(11.27s)이 2번째 요청(6.64s)보다 느린 것은 CUDA가 이 프로세스에서 특정 배치 크기 연산을 처음 실행할 때 알고리즘 탐색(cold start)을 하기 때문으로 추정 — 아래 "알려진 한계"의 명시적 워밍업으로 개선 여지 있음.

**알려진 한계 / 다음에 고려할 것**:
- `ThreadPoolExecutor(max_workers=1)`로 GPU 요청을 직렬화함 — 동시 요청 처리(여러
  위원이 동시에 말하는 시나리오)가 필요해지면 별도 설계 필요
- 첫 배치의 CUDA 알고리즘 탐색(cold start)에 대한 명시적 워밍업은 아직 안 함 — 배치
  루프 첫 진입 시 약간의 지연이 남아있을 수 있음
- 서버가 한번 뜨면 GPU 메모리에 모델이 계속 상주 — VRAM 여유 있는 A100이라 문제
  없을 것으로 예상하나 다른 GPU 작업과 동시 실행 시 확인 필요

## 시도 6 — 청크 단위 mp4 → MSE 연속 스트리밍 전환

지연 시간과는 별개로, 재생 자체가 "1초씩 끊어서" 재생되는 문제가 있었다. 원인: 프레임을
1초 분량씩 잘라 **독립된 mp4 파일**을 여러 개 만들고, 프론트엔드에서 `<video>.src`를
매번 새 blob URL로 갈아 끼우는 방식이었기 때문 — 청크 경계마다 비디오 디코더가 통째로
재시작되면서 끊김이 발생했다. 2차 프로젝트(`model_test.md`의 "프론트엔드 — 응답 영상
재생(MSE)" 섹션)는 처음부터 `MediaSource` + `sb.appendBuffer()`로 하나의 연속된 스트림에
이어붙이는 방식을 썼는데, 우리는 그렇지 않았다.

**조치 1**: `/generate-stream`의 청크 인코딩 로직을 통째로 교체.
- 기존: 프레임 24장(약 1초)씩 모아서 ffmpeg를 **3번**(비디오 인코딩 → 오디오 슬라이스 →
  mux) 실행해 완결된 mp4 파일을 만들고, 그 파일 전체를 WebSocket으로 전송 — 청크마다
  완전히 독립된 mp4
- 변경: 프레임이 하나씩 생기는 대로 ffmpeg muxing 프로세스(`-f image2pipe`)의 stdin에
  순서대로 흘려보내고, `-movflags frag_keyframe+empty_moov+default_base_moof`로 실시간
  생성되는 fragmented MP4 바이트를 stdout에서 읽어 그대로 WebSocket으로 전달. 키프레임
  간격을 `-g`/`-keyint_min`으로 fps(약 1초)에 고정해 fragment 크기를 적당히 유지
- 프론트엔드(`web_test_stream_mse.html`, 신규): `MediaSource` + `SourceBuffer.appendBuffer()`로
  도착하는 바이트를 그대로 이어붙임 (`sourceBuffer.mode = "sequence"`). 코덱 문자열
  (`avc1.42E01E, mp4a.40.2` = H.264 baseline level 3.0 + AAC-LC)이 ffmpeg의
  `-profile:v baseline -level 3.0`과 반드시 일치해야 브라우저가 받아들임

**버그 1(실측으로 발견)**: 1차 구현에서 muxer(ffmpeg) 프로세스 시작 후 첫 스트림 데이터가
나올 때까지 43.44s가 걸렸고, 그 직후 0.41s 만에 "전체 완료"가 떠서 **사실상 끝날 때까지
아무것도 안 흘려보내고 한 번에 쏟아냈다는 게 드러남**. 원인: mp4/mov muxer가 패킷을
파이프에 즉시 flush하지 않고 내부 버퍼링함. `-flush_packets 1`, `-max_interleave_delta 0`
추가로 해결.

**버그 2(실측으로 발견)**: 위 수정 후에도 첫 데이터까지 걸리는 시간이 요청마다 5.31s ~
36.35s로 들쭉날쭉했다. 원인 진단: 프레임을 ffmpeg stdin에 흘려보내는 `feed_frames()`가
asyncio 이벤트 루프에서 0.1초 폴링으로 동작했는데, GPU 추론(별도 스레드)이 CPU 작업
(PIL 기반 블렌딩 `get_image()`)으로 GIL을 오래 붙잡으면 이벤트 루프가 스케줄링을 못
받아 몇 초~몇십 초씩 멈추는 현상이었음(모델 로드/추론 스레드와 asyncio 이벤트 루프가
같은 GIL을 두고 경쟁). 2차 프로젝트도 같은 종류의 문제("1080p `add/` 아바타 전환 이후"
섹션의 "CPU 블렌딩 병목")를 겪었었음.

**조치 2**: 폴링 방식을 제거하고, 프레임이 생성되는 그 스레드 안에서 즉시 동기적으로
muxer stdin에 쓰도록 변경(`run_inference_sync`에 `frame_callback` 매개변수 추가).
출력 읽기는 전용 OS 스레드가 블로킹 `read()`로 큐에 밀어넣고, WebSocket 핸들러는 그
큐를 소비만 함 — 이벤트 루프 스케줄링에 의존하지 않는 구조로 변경.

**실측 결과(조치 2 이후)**: 첫 스트림 데이터 12.89s로 안정적으로 개선(이전처럼 널뛰지
않음). 단, 재생 자체는 여전히 버퍼링됨 — 아래 "시도 7" 참고.

## 시도 7 (완료) — 생성 속도 자체가 재생 속도보다 느린 문제

시도 6 이후에도 재생 중 버퍼링이 남아있어 재측정. 배치 36개(batch_size=8) 요청에서
`추론 시작 4.83s` → `전체 완료 45.53s`(약 40.7초 소요), 출력 프레임 최대 288장
(fps≈24 기준 오디오 길이 약 12초).

**즉 12초짜리 응답을 만드는 데 40.7초가 걸림 — 실제 재생 속도의 약 0.3배속.** 첫 데이터가
빨리 나와도(12.89s), 재생이 시작되면 금방 생성 속도를 따라잡아버려서 이후 계속 멈췄다
가기를 반복하는 구조적 문제. "시작 지연"이 아니라 "지속 가능한 처리량" 문제라 지금까지의
최적화(캐싱, 모델 상주, 스트리밍 구조 개선)로는 해결이 안 됨.

**2차 프로젝트(RTX 4060)와 비교**:

| | 2차 프로젝트 (RTX 4060) | 지금 (A100) |
|---|---|---|
| VAE 디코딩 | `_decode_fast`로 50%만 디코딩+보간 | 매 프레임 풀 디코딩 |
| 블렌딩 | `_blend_fast`(numpy, crop 영역만) | `get_image`(PIL, 풀프레임 변환) — CPU 병목 |
| 배치 크기 | 16 | 8 |
| GPU 워밍업 | 여러 배치 크기로 서버 시작 시 사전 워밍업 | 없음 |
| 결과 | 재생 예산 대비 약 0.9~1.1배속(거의 실시간) | 약 0.3배속 |

A100이 4060보다 원래 훨씬 빠른 하드웨어인데도 이 결과인 건, 2차 프로젝트가 썼던
최적화(특히 CPU 블렌딩 비용 절감)를 아직 하나도 반영 안 했기 때문으로 판단.

**다음 조치(예정)**: `musetalk/utils/blending.py`의 `get_image()` 실제 구현을 확인해서,
2차 프로젝트의 `_blend_fast()`(PIL 변환 없이 crop_box 영역만 numpy로 합성)에 해당하는
PIL-free 버전을 우리 `get_image()` 시그니처에 맞게 새로 작성해 교체 예정. 배치 크기 확대,
`_decode_fast`류 50% 디코딩은 필요시 후속 검토.

## 시도 8 — 마스크 사전계산 + PIL-free 블렌딩 + rawvideo 파이프 (성공, 실측 완료)

시도 7에서 진단한 "생성 속도가 재생 속도보다 느림(약 0.3배속)" 문제의 실제 원인을
찾기 위해, 먼저 배치 크기를 8→16으로 늘려봤다.

**배치 크기 확대 — 효과 없음**: 프레임당 처리 시간이 배치 크기와 무관하게 약
0.082~0.083s로 그대로. 이건 GPU 배칭 효율 문제가 아니라 **프레임당 CPU 비용이
고정 병목**이라는 뜻 — GPU를 더 굴려도 소용없다는 게 확인됨.

**원인 특정**: MuseTalk 실제 소스(`musetalk/utils/blending.py`)를 직접 받아서
확인한 결과, 지금 쓰고 있던 `get_image()`가 매 프레임마다 **얼굴 파싱(FaceParsing)
신경망 forward pass를 다시 실행**하고 있었다. 이 파일은 함수가 3개로 분리되어
있는데:
- `get_image()`: 신경망 파싱 + PIL 블렌딩을 전부 포함한 풀 파이프라인 (지금까지 이걸 매 프레임 호출 중이었음)
- `get_image_prepare_material()`: 마스크(`mask_array`)+`crop_box` 계산만, **얼굴
  위치가 고정이면 한 번만 계산하면 됨**
- `get_image_blending()`: 미리 계산된 마스크로 PIL 기반 합성만 하는 가벼운 버전

**조치 1 — 마스크 사전계산**: `_prepare_persona()`에서 프레임 위치별로
`get_image_prepare_material()`을 한 번만 호출해 `mask_list`/`crop_box_list`를
계산하고, 디스크 캐시(`results/{영상이름}_full_prep2_v15_em15.pt` — 스키마가
바뀌어 캐시 파일명에 `2`를 붙여 이전 캐시와 구분)에 함께 저장. `run_inference_sync()`는
매 프레임 `get_image()` 대신 `get_image_blending()`을 호출하도록 변경.

**조치 2 — PIL-free 블렌딩(`_blend_fast`)**: 마스크를 재사용해도 `get_image_blending()`은
여전히 매 프레임 720x1280 원본 전체를 PIL Image로 변환했다가 되돌리는 왕복을 함 —
실제로 바뀌는 건 `crop_box`(얼굴 주변 작은 영역)뿐인데도. 2차 프로젝트의
`_blend_fast` 패턴(numpy로 crop 영역만 직접 합성)을 이 프로젝트의 함수 시그니처에
맞게 새로 작성해 교체(그대로 복사 불가 — 원본이 시그니처가 달라서 새로 구현).
`copy.deepcopy(ori_frame)` 호출도 제거(`_blend_fast` 내부에서 `image.copy()`로 대체 — 훨씬 빠른 numpy 복사).

**조치 3 — PNG 인코딩/디코딩 왕복 제거**: 프레임을 ffmpeg에 `image2pipe`(PNG)로
넘기고 있었는데, ffmpeg가 어차피 바로 H.264로 재인코딩하므로 PNG 압축이 순수
낭비였음. `combine_frame.tobytes()`를 압축 없이 그대로 `rawvideo`(`-pix_fmt bgr24`)로
파이프에 흘려보내도록 변경. ffmpeg가 프레임 크기를 알아야 해서 `get_frame_size()`
(ffprobe로 W/H 조회) 헬퍼 추가, `-f rawvideo -pix_fmt bgr24 -s {W}x{H}`로 입력 지정.

**실측 결과**: 조치 1~3을 거치며 생성 속도가 약 0.3배속 → (마스크 캐싱+`_blend_fast`만)
약 0.8배속 → (rawvideo 적용 후) **약 1.6~1.8배속**까지 상승. 가장 최근 재측정
기준 16프레임 배치가 약 0.35~0.36초에 처리됨(17배치=272프레임/6.32s,
22배치=352프레임/8s) — 세션 전체에서 가장 큰 단일 개선.

## 시도 9 (완료) — 서버 사이드 프레임 페이싱, 그러나 새로운 버퍼링 패턴 발견

시도 8로 생성 속도가 실시간보다 빨라졌는데도(1.6~1.8배속), 실제 재생은 여전히
`waiting`/`playing`이 반복되며 끊겼다. 원인: GPU가 배치(16프레임) 단위로 몰아서
끝나기 때문에 **평균 속도는 빨라도 순간 전달 속도는 여전히 버스트(burst)** —
클라이언트가 아주 살짝만 전달이 늦어도 재생 버퍼가 바닥남.

**조치**: 프레임 *생성*(빠르게, `frame_queue`에 적재)과 ffmpeg stdin으로의
*전달*을 분리하는 전용 페이서(pacer) 스레드 추가. `_pace_and_feed()`가
`1/FPS` 간격으로 큐에서 하나씩 꺼내 ffmpeg에 씀 — 실시간 스트리밍 서비스들이
쓰는 "소스 측 페이싱" 패턴.

**실측 결과(문제 발견)**: 페이싱 적용 후 웹 로그를 보면 `playing` → 정확히
2초 후(`RESUME_THRESHOLD`) `버퍼 부족(0.07~0.08s)` → 다시 `playing` 패턴이
스트림이 끝날 때까지(약 47초 동안) **6초 주기로 11번 반복**됨. `[TIMING] 전체
완료`도 16.37s/18.44s로 늘어남(생성 자체는 6~8s에 끝났는데, 페이서가 나머지
시간을 실시간 속도로 다 채워야 하기 때문).

**원인**: 페이서가 `1/FPS`로 **정확히 1.0배속(재생 속도와 동일)** 으로만
전달하도록 고정돼 있어서, 시도 8에서 어렵게 확보한 1.6~1.8배속의 "여유분"을
그대로 버리고 있었음. 전달 속도 = 재생 속도이니 재생이 시작돼도 버퍼가 전혀
안 늘어나고, 재생하는 2초 동안 그대로 소진되어 다시 멈추는 게 반복됨.

**`model_test.md`(2차 프로젝트) 대조 확인**: 2차 프로젝트는 애초에 생성 속도가
**0.75배속(실시간보다 느림)** 이라 소스 쪽 페이싱을 아예 쓰지 않고, 순수
클라이언트 버퍼 임계값(`RESUME_THRESHOLD`/`PAUSE_THRESHOLD`)만으로 처리했다.
지금 프로젝트는 반대로 생성이 실시간보다 빠른 상황이라, 2차 프로젝트 방식을
그대로 가져올 수 없고 페이싱 배속 자체를 다르게 잡아야 한다는 게 확인됨.

**조치**: 페이싱 간격을 `1/FPS`가 아니라 `1/(FPS × PACE_SPEED_MULTIPLIER)`
(`PACE_SPEED_MULTIPLIER = 1.5`)로 재생 속도보다 살짝 빠르게 잡아서, 버스트는
완화하되 생성 속도의 여유분 일부가 버퍼로 계속 쌓이도록 조정.

**실측 결과(예상과 다름)**: 배속을 1.0→1.5로 올렸는데도 정지 패턴이 이전과
거의 동일하게 반복됨(약 6~7초 주기). 진단을 위해 WebSocket 전송 구간에
`[SEND-TIMING]` 계측(청크별 `wait`=ffmpeg가 다음 데이터를 만들 때까지 대기한
시간, `send`=`ws.send_bytes()` 자체가 걸린 시간)을 추가해서 재측정한 결과,
**`send`가 항상 0.0s** — 서버 프로세스 안에서는 전송이 전혀 느리지 않았음.
반면 서버가 측정한 `전체 완료`(13.79s)와 클라이언트가 실제로 마지막 데이터를
받은 시각(41.9s) 사이에 약 28초 차이가 있었음.

**결론**: 병목은 서버 코드(생성 속도, 페이싱, `ws.send_bytes`)가 아니라
**그 바깥 — 배포에 쓰던 Cloudflare Quick Tunnel(무료, 계정 없는 임시 터널,
Cloudflare가 공식적으로 대역폭/업타임 보장을 안 한다고 명시)의 실제 네트워크
전송 구간**이었음. `model_test.md`를 다시 확인해보니 2차 프로젝트는 이런
페이싱 자체가 필요 없었는데, 이유는 그쪽 생성 속도가 애초에 실시간보다
느렸기(0.75배속) 때문 — 우리처럼 대역폭이 좁은 무료 터널을 썼다면 2차
프로젝트도 같은 문제를 겪었을 것으로 추정되나, 문서에 해당 언급은 없음.

**검증 실험**: 인코딩 비트레이트를 낮춰서(720x1280/crf20 → 480x854/crf28,
전송량 3.85MB → 838KB, 약 4.6배 감소) 같은 요청을 재현한 결과 **버퍼링이
완전히 사라짐** — 대역폭 병목 가설이 확정됨.

## 시도 10 — 화질/버퍼 임계값 절충점 탐색 (완료)

시도 9에서 확정된 "네트워크(Cloudflare Quick Tunnel) 대역폭이 병목"이라는
결론을 바탕으로, 화질을 단계적으로 올리며 안정성이 유지되는 절충점을 탐색.

| 해상도/crf | 총 전송량(실측) | 버퍼링 |
|---|---|---|
| 720x1280 / crf20 (최초값) | 3.85MB | 반복(6~7초 주기) |
| 480x854 / crf28 | 838KB~1.14MB | **0회** |
| 576x1024 / crf24 | 1.14~1.57MB | 짧게 1회 |

→ **480x854 / crf28**을 최종값으로 채택(화질보다 안정성 우선, 재생 안정성이
검증된 값). Named Cloudflare Tunnel(도메인 필요) · ngrok 등 대안 터널도
검토했으나, 무료로 안정적인 대체 도메인/터널을 구하기 어려워 일단 Quick
Tunnel 유지 + 화질 하향으로 대응하기로 함.

**프론트엔드(`web_test_stream_mse.html`) 버퍼 임계값 조정**: 480x854/crf28
기준으로 `RESUME_THRESHOLD`(재생 시작/재개에 필요한 버퍼 여유)를 2.0초 →
0.3초로 낮춰봤더니 오히려 버퍼링이 훨씬 잦아짐(11초 동안 7회) — 너무 낮으면
네트워크의 순간적인 편차를 못 버틴다는 게 확인됨. `PAUSE_THRESHOLD`(0.15초)
보다는 충분히 여유 있는 **0.8초**로 재조정한 결과, 초반 대기 2.0초→0.7~0.8초로
줄면서도 실제 위원 발언 분량(약 22초 응답) 재생 내내 버퍼링 0회를 확인.

**부수적으로 발견/수정한 프론트엔드 버그**:
- 새로고침 없이 "시작" 버튼을 반복 클릭하면 `videoEl`(페이지에 고정된 하나의
  엘리먼트)에 `error`/`stalled`/`waiting`/`playing`/`ended` 리스너가 클릭할
  때마다 계속 누적 등록되어, 이후 모든 video 이벤트가 클릭 횟수만큼 중복
  로깅됨(실측: 7번 테스트 후 모든 이벤트가 정확히 7번씩 찍힘). 리스너 등록을
  클릭 핸들러 밖으로 옮겨 페이지 로드 시 한 번만 등록되도록 수정.
- `mediaSource.readyState`가 `"open"`이 아닐 때 `endOfStream()`을 호출해서
  나는 `InvalidStateError`가 간헐적으로 발생 — `readyState === "open"` 가드를
  추가했으나 완전히 없어지지는 않음(브라우저 내부 타이밍 문제로 추정, 재생
  자체엔 영향 없음을 반복 확인). 이 특정 에러만 조용히 무시하도록 처리.

**최종 설정**: 해상도 480x854, `crf 28`, `PACE_SPEED_MULTIPLIER 1.5`,
`RESUME_THRESHOLD 0.8`초, `PAUSE_THRESHOLD 0.15`초. 이 조합으로 실제 위원
발언 분량 텍스트 기준 초반 대기 0.7~0.8초, 재생 중 버퍼링 0회 달성.

## TensorRT를 시도하지 않은 이유

2차 프로젝트(`model_test.md`)에서 TensorRT FP16/FP32를 모두 시도했다가 포기했다:
FP16 TRT는 LayerNorm이 FP16 ForeignNode로 퓨즈되며 NaN이 발생했고(정밀도 문제라 GPU를
바꿔도 재현될 가능성이 높음), FP32 TRT는 RTX 4060에서 60~150배 느렸다(단, A100은
FP32/TF32 처리량이 훨씬 커서 이 결론이 안 바뀔 수도 있음 — 미검증). 지금 단계에서는
subprocess 구조 문제가 훨씬 큰 병목이라, TensorRT는 우선순위에서 제외.

|---|---|
| 초기 상태 | ~74s |
| 배치 인터리브 (효과 미미) | ~70s |
| + 페르소나 준비 캐싱 | ~40.5s |
| + 지연 import | ~33~35s |
| + 모델 상주 구조 (첫 요청) | 11.27s |
| + 모델 상주 구조 (워밍업된 요청) | 6.64s |
| + MSE 연속 스트리밍 전환 | (첫 청크는 비슷, 재생 끊김 문제로 전환) |
| + 마스크 캐싱 + `_blend_fast` + rawvideo | 생성 속도 0.3배속 → **1.6~1.8배속** |
| + 화질/버퍼 임계값 절충(시도 10) | **버퍼링 0회, 초반 대기 0.7~0.8초** |

초기 상태(첫 청크 ~74s) 대비 약 **11~12배** 단축, 생성 속도는 실시간 대비
**약 5~6배** 개선(0.3배속 → 1.6~1.8배속), 최종적으로 실사용 분량 재생에서
버퍼링 없이 안정적으로 동작하는 상태에 도달.

## 다음에 볼 것 (우선순위 낮음, 필요시)

- 서버 기동 직후 첫 요청이 2번째보다 느린 현상(11.27s vs 6.64s) — 배치 크기별 더미 추론으로
  명시적 워밍업하면 첫 요청부터 6~7초대로 맞출 수 있을 것으로 예상 (2차 프로젝트의
  `_warmup_gpu`와 같은 방식)
- TTS(OpenAI API) 호출 자체가 2.7~4.6초로 이제는 상대적으로 큰 비중 — 네트워크 지연이라
  서버 쪽에서 손댈 여지는 적음
- `ThreadPoolExecutor(max_workers=1)` 직렬화 — 동시 다중 요청 처리가 필요해지면 재검토
- Cloudflare Quick Tunnel은 대역폭 보장이 없는 임시/실험용 서비스라, 실제 배포
  단계에서는 Named Cloudflare Tunnel(도메인 필요) 등 안정적인 경로로 교체 검토 필요
  (지금은 무료로 안정적인 대체 도메인을 구하기 어려워 화질 하향으로 대응)
- `mediaSource.endOfStream()` 호출 시 간헐적으로 뜨는 `InvalidStateError`의
  근본 원인은 아직 못 찾음(재생 자체엔 영향 없어 조용히 무시 처리만 해둠)
