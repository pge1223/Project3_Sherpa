## 1. 전체 구조 (3단)

```
[프론트(React)] ──WebSocket──▶ [백엔드(FastAPI, 순수 중계)] ──WebSocket──▶ [코랩 서버(FastAPI+GPU)]
  CommitteeVideoStage.jsx         media.py /media/stream          app.py /generate-stream
                                                                    (cloudflared 터널로 외부 노출)
```

- **프론트**: 위원별로 WebSocket을 열고 발화 요청을 보낸 뒤, 돌아오는 바이너리 영상 데이터를 MediaSource로 이어붙여 실시간 재생한다.
- **백엔드(`media.py`)**: 아무 연산도 하지 않는다. 프론트가 보낸 JSON을 그대로 코랩으로 전달하고, 코랩이 보내는 상태 메시지/바이너리를 그대로 프론트로 돌려주는 순수 중계(relay)다. 코랩 URL(`MEDIA_SERVICE_WS_URL`, 재기동마다 바뀜)을 한 곳에서만 관리하려는 목적.
- **코랩(`app.py`)**: 실제 연산(TTS+MuseTalk GPU 립싱크 생성)이 일어나는 곳. 모델은 서버 시작 시 한 번만 GPU에 로드해서 상주시킨다.

## 2. 요청 포맷

프론트가 `/media/stream` WebSocket에 접속 직후 보내는 JSON (`ws.onopen`):

```json
{
  "speaker_id": "business_strategy",   // 코랩 PERSONA_MAP 키 (실제 위원 ID 아님, 슬롯 ID)
  "speaker_name": "사업전략 전문가",     // 화면 표시용 실제 위원 이름
  "order": 1,
  "text": "발화할 텍스트",
  "emotion": "neutral",                // 지금은 고정값, 실제 계산 로직 미구현
  "gesture_index": 1                   // 0=대기루프(rf), 1/2=제스처 영상. 직전 값 제외 랜덤 선택
}
```

## 3. 코랩 서버가 돌아가는 순서

1. 요청 수신 + fps/해상도 확인(ffprobe, 즉시)
2. ElevenLabs TTS 호출 → mp3 받아서 wav로 변환 (실측 7~12초, 텍스트 길이에 비례) → `status: audio_ready` 전송
3. ffmpeg muxer 프로세스 기동(원시 프레임을 실시간으로 mp4 조각으로 인코딩)
4. MuseTalk GPU 추론 시작 — Whisper로 오디오 특징 추출 → UNet/VAE로 배치(16프레임)씩 립싱크 프레임 생성 → 생성되는 즉시 muxer로 전달. **GPU 워커가 서버 전체에 1개뿔이라 다른 요청과 절대 동시 실행 안 됨**(직렬화) — VRAM 부족 걱정은 없지만 여러 위원 요청이 겹치면 뒤 요청은 GPU 순서를 기다린다.
5. 발화 프레임이 다 끝나면, 원본 루프 영상의 나머지 구간(GPU 연산 없이 원본 프레임 그대로)을 이어붙여 전체 길이를 루프 배수로 맞춘다("무음 tail"). tail 시작 시점에 `tts_end` 전송(아래 4-2 참고).
6. 다 보내면 `done` 전송, 연결 정리.

프레임은 완성될 때마다 즉시(파일 다 만들 때까지 기다리지 않고) WebSocket으로 흘려보낸다 — 이게 "실시간 스트리밍"의 핵심.

## 4. 프론트(`CommitteeVideoStage.jsx`) 핵심 설계

### 4-1. 두 레이어 구조 (idle / stream)

위원 타일마다 `<video>`를 **두 개** 둔다.
- **idle**: 마운트 시 딱 한 번 재생 시작하면 이후 `src`도 `play()`도 절대 다시 안 건드림. 컴포넌트 사는 내내 뒤에서 계속 돈다.
- **stream**: 평소 투명(opacity:0)하게 숨겨져 있다가, 실제 발화 데이터가 버퍼에 충분히 쌓이면(`RESUME_THRESHOLD = 1.5`초) 위로 드러난다(opacity:1).

**이유**: 원래는 태그 하나를 재사용(idle↔stream 전환마다 `src` 교체)했는데, 이 방식이 세 가지 버그의 공통 원인이었다:
- 스트림 버퍼 부족으로 잠깐 멈추면 화면 전체가 얼어붙어 보임
- 스트림 끝나고 idle로 복귀 시 `video.play()`를 다시 불러야 하는데, **크롬이 "전력 절약을 위해 배경 영상 재생을 중단시켰다"며 재생 요청 자체를 거부**하는 경우가 실측으로 확인됨(한 번은 147초간 안 풀림)
- 버퍼링 중 화면이 멈춘 것처럼 보임

태그를 분리하고 idle을 아예 안 건드리면 세 문제가 동시에 해결된다.

### 4-2. MediaSource 스트리밍 재생

- `MIME_CODEC = 'video/mp4; codecs="avc1.42E01E, mp4a.40.2"'` — 코랩 ffmpeg 인코딩 설정(`-profile:v baseline -level 3.0`)과 반드시 일치해야 함.
- 바이너리 프레임이 오면 `SourceBuffer.appendBuffer()`로 계속 이어붙인다.
- `RESUME_THRESHOLD = 1.5`, `PAUSE_THRESHOLD = 0.15` — 재생 시작/일시정지 버퍼 임계값. tail 프레임을 페이싱 없이 몰아서 보내도록 코랩을 고친 뒤(아래 5-1) 값이 0.8→1.5로 상향됨(재생을 살짝 늦게 시작하는 대신 시작 후 안 끊기게).

### 4-3. 위원별 상태는 전부 개별 관리

`tokensRef`, `monitorTimersRef`, `activeCleanupsRef`, `lastGestureIndexRef` 등을 전부 `speakerId`를 키로 하는 Map으로 관리한다. 위원 A가 재생 중이고 B가 백그라운드에서 준비 중일 때 서로의 상태를 침범하면 안 되기 때문(초기엔 공유 변수 하나로 짜서 3~4명일 때만 재현되는 경합 버그를 겪음, 아래 6장).

### 4-4. 위원 전환 파이프라이닝 + 전환 게이트

- **문제**: 위원 A 발언 끝나고 B로 넘어가는 사이가 너무 오래 걸림.
- **1차 개선**: A가 화면에 뜨는 순간(reveal) B의 요청을 미리 보내 백그라운드에서 준비시킴(TTS는 GPU를 안 써서 A의 GPU 생성 중에도 병렬 가능). 다만 화면 노출(reveal)은 A가 완전히 끝날 때까지 보류.
- **2차 개선**: "A의 무음 tail이 긴 경우, B가 이미 준비됐는데도 의미 없는 tail을 계속 기다리는" 낭비 발견. tail은 원래 "발화 끝나고 대기 루프로 자연스럽게 이어지도록" 만든 건데, 지금 두 레이어 구조에서는 idle이 이미 독립적으로 돌고 있어서 이 정합성 자체가 무의미해졌다는 걸 재확인. **하지만 tail을 강제로 자르진 않기로 함**(원본 자연 재생 유지, 오디오 안 겹치는 게 확인됨 — tail은 무음이라 GPU 연산도 없음).
- **최종 설계**: 코랩이 TTS(실제 발화) 종료 시점에 `tts_end` 신호(+`speech_seconds`: 몇 초 지점부터 tail인지)를 보내면, 프론트는 `video.currentTime`이 그 지점을 실제로 지날 때만(신호 도착 즉시 X — 데이터가 항상 재생 위치보다 앞서 도착하므로 시차 있음) "발언 중" 배지를 끄고, 그로부터 **+2초 뒤(또는 그 전에 자연 종료되면 즉시, 둘 중 먼저 오는 조건)** 다음 위원이 화면을 넘겨받는다.
- `transitionGateRef = { readyAt, forceReady, owner }` — 다음 위원이 화면을 넘겨받아도 되는 시점을 표현하는 단일 게이트. `owner`는 "지금 이 게이트를 쥔 위원이 누구인지" — 이게 없으면 늦게 끝나는 위원의 뒤늦은 정리 로직이 이미 넘어가 있는 게이트를 멋대로 열어버리는 버그가 남(아래 6장).
- "화면에 한 번에 한 명만 노출"이라는 예전 제약은 없앰 — `revealedMap`(stream 영상 노출 여부)은 위원별로 독립, `speakingMap`("발언 중" 배지)만 한 번에 하나. 위원마다 타일이 분리돼 있어 A가 무음 tail 재생 중이어도 B가 이미 발화를 시작할 수 있음.

### 4-5. gesture_index 로테이션

위원 1명당 영상이 3개(rf + 제스처 2개)라는 전제(수동 동기화 필요, `AVATAR_SLOTS`와 마찬가지). 매 발화마다 직전에 그 위원이 썼던 인덱스만 제외하고 나머지 중 랜덤 선택(`pickGestureIndex`) — 같은 자세로만 계속 말하지 않도록.

### 4-6. stream→idle 전환 이음매

idle은 마운트 이후 한 번도 안 건드리고 독자적으로 계속 돌아서, stream이 끝나는 순간 idle이 몇 프레임째인지가 우연에 달려있어 자세가 툭 튀어 보이는 문제가 있었다. `finish()` 시점에 idle을 0번 프레임으로 `currentTime = 0` seek하되, `seeked` 이벤트(또는 300ms 타임아웃 안전장치)를 기다린 뒤에야 화면을 전환한다 — 그동안 stream은 이미 멈춘 마지막 프레임을 계속 보여주고 있어서 화면이 비는 순간이 없다.

## 5. 겪은 버그와 해결 히스토리 (요약)

| 버그 | 원인 | 해결 |
|---|---|---|
| 짧은 발화에서 앞부분 정체/버퍼링 | tail(무음) 프레임까지 실시간 배속으로 페이싱하며 전송 | tail은 페이싱 없이 즉시 전송(`_pace_and_feed`의 `is_tail` 분기) |
| 대기 루프 복귀 시 화면 정지 | 크롬이 반복되는 `video.play()` 재호출을 전력절약 정책으로 거부 | 두 레이어 구조로 idle을 아예 재호출 안 함 |
| 발화 안 끝났는데 배지 꺼지고 다음 위원 등장 | `tts_end` 신호를 "도착 즉시" 반영 — 데이터가 재생 위치보다 항상 앞서 도착 | 신호는 값만 저장, `video.currentTime`이 실제로 그 지점 지날 때만 반영 |
| C 발언중인데 D가 먼저 시작(4명 이상에서만 재현) | 전환 게이트에 소유자 개념이 없어서, 늦게 끝난 위원의 정리 로직이 이미 넘어간 게이트를 덮어씀 | 게이트에 `owner` 추가, "지금 게이트 주인 본인만" 열고 닫게 제한 |
| 발화 도중 연결이 끊기며 다음 위원이 끼어듦 | 백엔드↔코랩 WebSocket keepalive 핑퐁 타임아웃(기본 20초) — 코랩이 GPU 작업으로 바빠지면 핑에 제때 응답 못 해 "죽었다"고 오판. **양방향**(백엔드→코랩, 코랩→백엔드) 각각 별도로 존재 | 백엔드: `websockets.connect(..., ping_interval=None)`. 코랩: uvicorn 실행 시 `--ws-ping-interval 86400 --ws-ping-timeout 86400` |
| stream→idle 전환 시 자세 튐 | idle 위치가 stream 종료 시점과 무관하게 독자적으로 흘러감 | 4-6 참고(seek + seeked 대기) |

## 6. 재사용 시 참고할 점

- 위원 1명짜리 스트리밍 재생만 필요하다면(이번 문서 하이라이트 기능처럼), `CommitteeVideoStage.jsx`의 상위 오케스트레이션(배치 큐, 여러 위원 간 전환 게이트)까지 통째로 가져올 필요는 없다 — 그건 "여러 위원이 순서대로 말한다"는 시나리오 전용 로직이다.
- 다만 하위 배관(WebSocket 연결 + MediaSource 버퍼링 + 두 레이어 idle/stream 전환 + `tts_end`/`done` 메시지 처리)은 위 5장의 버그들을 다 겪고 나서야 안정화된 코드라, **새 화면에서 처음부터 다시 짜지 말고 재사용/추출해서 쓰는 걸 강력 추천**.
- `RESUME_THRESHOLD`/`PAUSE_THRESHOLD`/`ping_interval` 관련 설정값은 코드에 왜 그 값인지 주석으로 남겨뒀으니 그대로 가져다 쓰면 됨.
