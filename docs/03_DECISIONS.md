# 03. Decisions

## 1. 확정 결정

### 서비스 방향 전환 (2026-07-20)

- 팀 논의를 거쳐 서비스 방향을 전환한다. 상세 근거와 화면·데이터 설계는
  `docs/REVIEW_BOARD_서비스_방향성_정리_20260720.md`를 우선한다.
- 기존 competition 도메인 고정 4위원 체계(창의성·기술·사업전략·완성도,
  `ai/meeting/personas/persona_cards.json`·`rubric_mapping_competition.json`)는
  더 이상 신규 개발에 쓰지 않는다. **코드·데이터는 삭제하지 않고 legacy로
  남긴다.**
- 사용자 진입 경로를 "작성 전(주제 발굴 회의) / 작성 후(문서 피드백)" 2-모드로
  나눈다. 작성 전 모드는 신규 플로우다.
- 작성 후(문서 피드백)의 기본값은 즉시 결과 제공이다. 기존처럼 업로드마다
  항상 위원별 독립 평가(배치) + 위원장 종합을 도는 구조가 아니라, AI 위원
  소집은 사용자가 필요할 때 선택하는 대화형으로 바뀐다.
- 작업 담당: 기획/개발 위원 페르소나 카드는 용준, 신규 프론트 플로우는 가은.
- 이 전환이 `contracts/`나 다른 담당자의 인터페이스에 영향을 주는 부분은
  이 문서의 "4. 결정 변경 절차"를 따라 별도로 확정한다(이 항목은 방향
  합의 기록이며, 공통 계약 변경 자체를 승인하는 것은 아니다).

### 개발 환경

- Python 3.12
- Conda 가상환경
- 환경명 권장값: `review-board`
- Python 패키지는 Conda 환경 안에서 requirements 기반 설치

### 기술 스택

- 프론트엔드: React
- 백엔드: FastAPI
- 데이터베이스: MongoDB
- Vector DB: 미확정. RAG 담당자가 후보를 비교하고 관련 담당자 협의 후 최종안을 제안
- RAG: LangChain
- 회의 오케스트레이션: LangGraph
- 임베딩: KURE-v1 우선
- 문서 파싱: PyMuPDF, python-docx, python-pptx
- AI 영상: TTS + MuseTalk
- 배포: NCP


### Vector DB 선정 상태 및 협의 범위

- 상태: 변경 제안 및 비교 검토 단계
- 주관 담당자: 용준
- 협의 담당자: 경이, 윤한. 프론트 계약 영향이 있을 경우 가은도 포함
- Vector DB 제품은 아직 프로젝트 공통 확정 사항이 아니다.
- 용준은 데이터 규모, 로컬·서버 운영 방식, 메타데이터 필터, 영속성, 성능, 팀 숙련도, NCP 배포 적합성을 비교해 최종안을 제안한다.
- 경이는 Retriever 요청·응답 JSON과 LangGraph 소비 구조에 미치는 영향을 검토한다.
- 윤한은 NCP 배포 방식, 서버 구성, 영속성, 백업, 모니터링과 비용 영향을 검토한다.
- 가은은 검색 상태, 출처 표시, 오류 응답 등 프론트 계약 변경이 있을 때 영향을 검토한다.
- 후보 예시는 Chroma, FAISS, Qdrant, Milvus, Weaviate, Elasticsearch/OpenSearch 계열 등이지만 특정 제품을 기본값으로 강제하지 않는다.
- 팀 동의 전에는 특정 제품을 확정 기술로 표현하지 않는다.
- 최종 선정 전에 Retriever 요청·응답 JSON과 저장소 추상화 인터페이스를 먼저 고정한다.
- 제품 차이는 RAG 모듈 내부로 최대한 캡슐화하고, 공통 계약 변경이 필요하면 정해진 변경 절차를 따른다.

### 기술 선택 권한 원칙

- 각 영역 담당자는 자신의 기술 후보 비교와 최종 제안을 주도한다.
- 선택 결과가 다른 담당자의 인터페이스, 데이터 구조, 배포 환경 또는 작업 범위에 영향을 주면 단독으로 확정하지 않는다.
- 영향받는 담당자와 사전 협의하고 팀 동의를 받은 뒤 최종 확정한다.
- 담당자의 전문적 결정 권한과 공통 구조에 대한 팀 합의를 함께 보장한다.

### 평가 정책

- 위원들은 독립 평가한다.
- 핵심 지적에는 원문 근거를 연결한다.
- 위원장은 새로운 근거를 생성하지 않는다.
- 최종 점수는 Python 규칙 엔진이 계산한다.
- 사용자에게 점수는 당선 확률이 아니라 평가기준 충족도로 표현한다.
- 근거 부족 시 추정하지 않고 부족 상태를 표시한다.

### 개발 방식

- 공통 JSON 계약과 Mock 데이터로 병렬 개발한다.
- 다른 담당자의 완성을 기다리지 않는다.
- Schema 변경 시 `schema_version`을 관리한다.
- 공통 계약 변경은 팀 동의 후 반영한다.

### 영상 정책

- MVP 영상 페르소나는 2명이다.
- 2분할 화상회의형 UI를 우선한다.
- 실존 인물을 모사하지 않는다.
- 영상 실패 시 정적 이미지 + TTS 폴백을 사용한다.
- AI 회의 시뮬레이션임을 표시한다.

## 2. 검토 중

- 최종 LLM 모델
- 모델별 온도와 토큰 설정
- KURE-v1 로컬 GPU 실행 방식
- Vector DB 제품 및 운영 방식
- 비동기 작업 도구 선택
- TTS 서비스 선택
- MuseTalk 실행 위치
- Object Storage 최종 사용 여부
- 인증 방식 세부 구현
- PDF 생성 라이브러리
- 프론트 상태관리 라이브러리
- 최종 Top-K와 검색 임계값
- 도메인별 Persona Card 세부 내용
- 평가항목 가중치 추출 방식

## 3. 하지 않기로 한 방향

- LLM이 최종 점수를 직접 결정하는 방식
- 근거 없이 공식 기준처럼 단정하는 방식
- 영상 생성 실패로 전체 평가 결과를 실패 처리하는 방식
- 업무를 순차 의존 구조로 진행하는 방식
- 다른 프로젝트의 스택과 요구사항을 자동 재사용하는 방식

## 4. 결정 변경 절차

1. 현재 결정과 문제를 설명한다.
2. 대안과 영향 범위를 정리한다.
3. 영향받는 담당자와 영향 범위를 확인한다.
4. 필요한 경우 변경 전후 JSON 예시와 대안을 공유한다.
5. 관련 담당자 협의와 팀 동의를 받는다.
6. 공통 계약 변경 시 `schema_version`을 갱신한다.
7. 문서, Mock, 테스트를 갱신한다.
8. 구현에 반영한다.
