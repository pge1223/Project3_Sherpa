# AI Review Board 병렬 개발 업무 흐름

## 1. 왜 병렬 개발로 진행하는가

기존 방식처럼 한 사람의 작업이 끝난 뒤 다음 사람이 이어받으면 아래와 같은 문제가 생긴다.

```text
RAG 완료
→ LangGraph 개발
→ 점수 및 결과 처리
→ 프론트 연결
→ 영상 연결
```

이 구조는 앞 단계가 지연되면 뒤 작업 전체가 멈추게 된다.

따라서 AI Review Board는 **공통 JSON 스키마와 Mock 데이터**를 먼저 확정한 뒤, 각 담당자가 실제 모듈 완성을 기다리지 않고 동시에 개발한다.

> 핵심 원칙  
> 각 담당자는 다른 담당자의 완성본이 아니라, 미리 정의된 입출력 규격을 기준으로 개발한다.

---

## 2. 최종 역할 분담

| 담당자 | 담당 영역 | 핵심 결과물 |
|---|---|---|
| 가은 | PM, React 프론트, 페르소나·프롬프트 기획 | 화면 흐름, Persona Card, 프롬프트 초안, React UI |
| 윤한 | FastAPI, MongoDB, CRUD, NCP | API, DB 스키마, 파일·영상 저장, 배포 환경 |
| 경이 | LangGraph, 점수 엔진, 평가 결과 구조 | 회의 워크플로, 평가 결과 JSON, 점수 계산 |
| 용준 | RAG 파이프라인 | 문서 파싱, 청킹, 임베딩, 검색, 출처 연결 |
| 재인 | AI 휴먼 영상 제작 | 페르소나 영상 2종, TTS, MuseTalk, 영상 폴백 |

## 3. 공통으로 먼저 확정할 것

모든 팀원이 동시에 개발하려면 아래 계약을 먼저 맞춰야 한다.

### 3.1 공통 식별자

```json
{
  "project_id": "project_001",
  "document_id": "document_001",
  "meeting_id": "meeting_001",
  "persona_id": "business_strategy"
}
```

### 3.2 핵심 데이터 계약

- 문서 업로드 결과
- RAG 검색 요청·응답
- 위원별 평가 결과
- 점수 계산 입력·출력
- 회의 최종 결과
- AI 영상 발언 스크립트
- 영상 생성 상태와 URL

공통 기준 파일:

```text
AI_Review_Board_meeting_contract_v1.json
```

이 파일의 필드명과 구조를 기준으로 개발하며, 변경이 필요한 경우 팀 공유 후 버전을 올린다.

```text
v1.0.0 → 필드 구조 최초 확정
v1.1.0 → 선택 필드 추가
v2.0.0 → 기존 필드명 또는 구조 변경
```

## 5. 담당자별 병렬 개발 방법

## 5.1 가은 — PM·프론트·페르소나/프롬프트 기획

### 실제 연동을 기다리지 않고 진행할 작업

- 서비스 화면 흐름 정의
- 문서 업로드 화면
- 분석 진행 상태 화면
- 2인 AI 회의 화면
- 현재 발언자 강조
- 자막 및 근거 패널
- 평가 리포트 화면
- Persona Card 작성
- 위원별 평가 관점과 말투 정의
- 위원별·위원장 프롬프트 초안 작성

### 사용하는 Mock 데이터

- 샘플 프로젝트
- 샘플 문서 처리 상태
- 샘플 위원 평가 결과
- 샘플 점수와 근거
- 샘플 영상 URL

### 다른 담당자에게 전달할 결과물

```text
화면 명세
Persona Card
Prompt Spec
API 요청·응답 화면 요구사항
영상 배치 및 재생 규칙
```

---

## 5.2 윤한 — FastAPI·MongoDB·CRUD·NCP

### 실제 RAG와 LangGraph를 기다리지 않고 진행할 작업

- MongoDB 컬렉션 설계
- 프로젝트 CRUD
- 문서 메타데이터 CRUD
- 회의 및 평가 결과 저장
- FastAPI 엔드포인트
- 파일 업로드와 저장
- 영상 URL 저장
- 작업 상태 관리
- NCP 배포 환경 구성

### 사용하는 Mock 데이터

- 샘플 RAG 결과
- 샘플 LangGraph 결과
- 샘플 점수 결과
- 샘플 영상 생성 결과

### 다른 담당자에게 제공할 결과물

```text
API 명세
MongoDB 스키마
파일·영상 저장 규칙
비동기 작업 상태 규격
배포 주소 및 환경변수 기준
```

---

## 5.3 경이 — LangGraph·점수 엔진·평가 결과 구조

### 실제 RAG를 기다리지 않고 진행할 작업

- LangGraph State 정의
- 회의 준비 노드
- 위원 A 독립 평가 노드
- 위원 B 독립 평가 노드
- 보완 의견 노드
- 점수 계산 노드
- 위원장 종합 노드
- 영상용 발언 데이터 생성 노드
- 평가 결과 JSON 검증
- Python 규칙 기반 점수 계산

### 사용하는 Mock 데이터

```json
{
  "query_id": "query_001",
  "results": [
    {
      "chunk_id": "chunk_001",
      "document_id": "document_001",
      "page": 4,
      "text": "시장성 평가 배점은 30점이다.",
      "score": 0.88
    }
  ]
}
```

### 가은에게 받는 것

- Persona Card
- 위원별 평가 관점
- 말투와 회의 순서
- 프롬프트 초안

### 다른 담당자에게 제공할 결과물

```text
LangGraph 워크플로
reviewer_result JSON
final_meeting_result JSON
score_summary JSON
media_script JSON
```

## 5.5 재인 — AI 휴먼 영상 제작

- 2분할 화상회의 화면 테스트
- 영상 해상도·길이·파일 형식 결정
- 영상 생성 실패 폴백 제작

{
  "meeting_id": "meeting_001",
  "speaker_id": "business_strategy",

## 6. 개발 시작 순서

## Step 1. 계약 확정

전원이 함께 아래 항목을 확인한다.

- ID 명명 규칙
- JSON 필드명
- API 경로
- 상태값
- 오류 코드
- Persona ID
- 문서 유형
- 회의 결과 구조
- 영상 생성 결과 구조

## Step 2. Mock 데이터 배포

공통 저장소에 아래 파일을 둔다.

```text
/mocks
 ├─ project.json
 ├─ document_status.json
 ├─ rag_response.json
 ├─ reviewer_result.json
 ├─ final_meeting_result.json
 ├─ media_script.json
 └─ media_job_result.json
```

## Step 3. 각자 독립 개발

각 담당자는 Mock 데이터를 이용하여 자신의 모듈을 완성한다.

## Step 4. 부분 통합

```text
용준 RAG ↔ 경이 LangGraph
경이 결과 ↔ 윤한 FastAPI·MongoDB
윤한 API ↔ 가은 React
경이 media_script ↔ 재인 영상
재인 video_url ↔ 윤한 저장 ↔ 가은 재생
```

## Step 5. 전체 통합

```text
문서 업로드
→ 파싱·RAG
→ LangGraph 회의
→ 점수 계산
→ 결과 저장
→ 영상 생성
→ React 회의 화면
→ 최종 리포트
```

---

## 7. 통합 체크포인트

### 체크포인트 1 — 규격 확인

- Mock JSON으로 모든 화면과 모듈이 동작하는가
- 필드명이 서로 일치하는가
- 필수값과 선택값이 명확한가

### 체크포인트 2 — 부분 연결

- RAG 결과를 LangGraph에서 읽을 수 있는가
- LangGraph 결과를 점수 엔진과 DB에서 처리할 수 있는가
- `media_script`를 영상 모듈이 처리할 수 있는가

### 체크포인트 3 — 전체 연결

- 사용자가 업로드한 문서가 최종 회의 화면과 리포트까지 연결되는가
- 핵심 의견에 출처가 표시되는가
- 영상 실패 시 정적 이미지+TTS로 대체되는가

---

## 8. 변경 관리 규칙

공통 JSON 구조를 임의로 바꾸면 다른 팀원의 작업이 깨질 수 있다.

변경이 필요할 때는 아래 순서를 따른다.

```text
변경 제안
→ 영향받는 담당자 확인
→ JSON 예시 공유
→ 팀 동의
→ schema_version 변경
→ Mock 파일 갱신
→ 코드 반영
```

필드 삭제나 이름 변경은 가급적 피하고, 필요한 경우 새로운 필드를 추가하는 방식으로 진행한다.

---

## 9. 핵심 정리

> 용준의 RAG가 완성될 때까지 경이가 기다리지 않는다.  
> 경이의 LangGraph가 완성될 때까지 윤한과 가은이 기다리지 않는다.  
> 실제 회의 문장이 완성될 때까지 재인이 기다리지 않는다.

모든 담당자는 **공통 계약과 Mock 데이터**로 먼저 개발하고, 마지막에 실제 모듈로 교체한다.

```text
Mock으로 병렬 개발
→ 모듈별 단위 테스트
→ 부분 통합
→ 실제 데이터 교체
→ 전체 통합
```
