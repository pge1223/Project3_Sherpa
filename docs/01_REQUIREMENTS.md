# 01. Requirements

## 1. 범위 구분

### 1차 MVP

- 사용자 로그인
- 프로젝트 생성 및 조회
- 평가 대상 문서 업로드
- 평가기준 문서 업로드
- PDF, DOCX, PPTX 파싱
- 문서 청킹 및 임베딩
- 선정된 Vector DB 기반 검색
- 도메인 자동 분류
- 도메인 수동 변경
- 도메인별 Persona Card
- 위원별 독립 평가
- 위원장 종합
- Python 규칙 기반 점수 계산
- 수정 우선순위 생성
- 회의 결과 저장
- 종합 리포트 및 위원별 상세 의견
- 평가 의견과 원문 출처 연결
- 핵심 AI 페르소나 2명의 TTS·MuseTalk 영상
- 2분할 화상회의형 재생 화면
- 영상 실패 시 정적 이미지 + TTS 폴백
- NCP 배포
- 처리 진행 상태 및 오류 표시

### 2차 범위

- 프로젝트 삭제
- 문서 미리보기
- 유사 성공 사례 검색
- 수정 전후 재평가 비교
- 결과 PDF 내보내기
- 점수 산정 근거 상세 분해
- 영상 품질과 배경 연출 개선

### 확장 범위

- 시장·정책 외부 자료 자동 수집
- 가상 사용자 페르소나 평가
- 특정 위원만 재평가
- AI 위원 3~5명 영상화
- 회의장 좌석형 화면
- 동적 그리드와 영상 캐시
- 관리자용 Persona Template 편집

## 2. 기능 요구사항

### 인증과 프로젝트

- 사용자는 로그인할 수 있어야 한다.
- 사용자는 프로젝트를 생성하고 목록과 상세를 조회할 수 있어야 한다.
- 프로젝트 접근 시 소유권을 검증해야 한다.

### 문서 처리

- PDF, DOCX, PPTX를 지원해야 한다.
- 파일 확장자, MIME, 크기를 검증해야 한다.
- 페이지·슬라이드·문단 메타데이터를 보존해야 한다.
- 파싱 실패와 임베딩 실패 상태를 사용자에게 표시해야 한다.

### RAG

- 문서를 의미 단위로 청킹해야 한다.
- KURE-v1 임베딩을 우선 사용한다.
- RAG 담당자가 선정한 Vector DB에 벡터와 메타데이터를 저장해야 한다.
- 위원 역할별 질의를 생성해야 한다.
- Top-K와 metadata filter를 지원해야 한다.
- 근거가 부족한 경우 이를 명시해야 한다.

### 평가와 회의

- 각 위원은 다른 위원의 결과를 보기 전에 독립 평가해야 한다.
- 위원별 결과는 공통 JSON 형식을 따라야 한다.
- 위원장은 기존 평가와 근거만 종합해야 한다.
- 최종 점수는 Python 규칙 엔진으로 계산해야 한다.
- 합의, 이견, 수정 우선순위를 제공해야 한다.

### 결과와 UI

- 종합 점수, 강점, 문제점, 수정 우선순위를 표시해야 한다.
- 위원별 상세 의견을 분리해서 볼 수 있어야 한다.
- 각 의견에서 원문 근거로 이동할 수 있어야 한다.
- 처리 상태와 실패 사유를 표시해야 한다.

### AI 영상

- 핵심 페르소나 2명을 영상화해야 한다.
- 발언 단위로 TTS와 립싱크 영상을 생성해야 한다.
- 발언 순서를 보장해야 한다.
- 영상 실패 시 이미지 + TTS로 대체해야 한다.
- 사용자에게 AI 회의 시뮬레이션임을 표시해야 한다.

## 3. 비기능 요구사항

- Python 3.12와 Conda를 사용한다.
- 환경변수로 비밀값을 관리한다.
- 공통 Schema에 `schema_version`을 포함한다.
- 동일 점수 입력에는 동일 결과가 나와야 한다.
- 공통 계약 변경은 영향받는 담당자 확인, JSON 예시 공유, 관련 담당자 협의와 팀 동의 후 반영한다.
- 각 모듈은 Mock 데이터로 단독 실행 가능해야 한다.
- 로그에는 프로젝트, 문서, 회의, Job 식별자를 포함한다.
- 일부 모듈 실패가 전체 프로젝트 데이터 손실로 이어지지 않아야 한다.

## 4. 제외 사항

- 실제 당선 확률 예측
- 근거 없는 시장성·정책성 판정
- 실존 인물 음성·얼굴 모사
- 영상 생성이 끝날 때까지 전체 리포트를 차단하는 구조
- LLM이 임의로 최종 점수를 확정하는 방식

## 5. 기술 선택 및 협의 요구사항

- 각 기술 영역의 주관 담당자는 후보 비교와 최종 제안을 주도해야 한다.
- 기술 선택이 다른 모듈의 인터페이스, 데이터 구조, 배포 환경 또는 작업 범위에 영향을 주는 경우 관련 담당자와 사전 협의해야 한다.
- 팀 동의 전에는 검토 중인 후보를 확정 기술로 문서화하지 않는다.
- Vector DB 선정 시 RAG 담당자는 최소한 LangGraph 담당자와 Retriever 계약 영향을, 백엔드·인프라 담당자와 NCP 배포·운영 영향을 검토해야 한다.
- 공통 JSON 계약이 변경되면 `schema_version`, Mock 데이터, 테스트와 관련 문서를 함께 갱신해야 한다.
