## 1. 수정본을 3번째 제출하면 버전이 v1.2로 안 쌓이고 v1.1이 덮어써진다

**증상**
- "다음 수정본 제출"을 반복하면 v1.0 → v1.1까지는 되는데, 3번째 제출부터 v1.2가 안 생김.
- 대신 최신본이 v1.1로 덮어써지고, 직전 v1.1은 v1.0으로 밀리며, **최초 v1.0 기록이 사라짐**.

**원인 (백엔드·프론트 양쪽 모두 2개로 제한)**
- 백엔드 `GET /projects/{id}/comparison` 이 **최근 2개 meeting만** 꺼내 `build_revision_comparison`
  (수정 전/후 2개 비교)에 넘김. 3개째부터는 오래된 게 조회에서 빠짐.
- 프론트 `buildVersionsFromComparison` 가 결과를 `[v1.0, v1.1]` **2개로 하드코딩**해 렌더.

**해결**
- `ai/meeting/scoring/comparison.py` 에 `build_version_history(documents)` 신설 — 프로젝트의
  **모든 meeting을 오래된→최신 순으로 v1.0, v1.1, v1.2 …** 누적 변환(항목별 점수·판정·소속위원·
  지적 + 직전 대비 new/resolved).
- `/comparison` 이 `versions` 배열 + 최신 pairwise `comparison`을 함께 반환하도록 교체.
- 프론트 `buildVersionsFromHistory(versions)` 로 히스토리 전체를 렌더(점수 추이 그래프에 제출한
  만큼 점 누적).

**검증**
- 기존 프로젝트 `/comparison` 이 v1.0(82)·v1.1(77)·v1.2(21.5) **3버전** 반환 확인.

**교훈**
- analyze는 매번 `meeting_repo.create`로 meeting을 **새로 쌓는다**(덮어쓰지 않음). 데이터는 이미
  누적돼 있었고, 버그는 순전히 "조회·렌더에서 2개로 잘라 본" 것. 누적 데이터를 다룰 땐 상한
  하드코딩(2개)을 의심할 것.

## 2. 같은 문서인데 채점 총점이 매번 다르게 나온다 (31.5 ↔ 64)

**증상**
- "완성 리포트에서 나쁜 문서 데이터 점수가 너무 후하다"는 지적으로 시작.
- 파보니 **같은 50점대 테스트 문서**를 **같은 프로젝트·같은 위원·같은 rubric**으로 돌렸는데
  총점이 **31.5 ↔ 64**(±32.5)로 튐. data_utilization 도 5/20 ↔ 15/20.
- 후하게 걸린 실행에서는 위원이 "구체적 데이터 출처 부족"이라고 **지적하면서도** acceptable(15/20)을
  줘서 [엄정 채점] 규칙을 무시.

**원인**
- `temperature=0` 은 이미 적용돼 있었지만(meetings.py llm_call), **gpt-4o-mini는 temperature=0
  으로도 완전 결정론이 아니다**. 특히 "주제는 맞지만 근거는 빈약한 **경계(borderline) 문서**"에서
  판정(needs_improvement ↔ acceptable)이 flip되고, 그게 5/20 ↔ 15/20 점수 격차로 증폭됨.
- 즉 겉보기 "너무 후함"의 **진짜 원인은 점수 흔들림**이었다(운 나쁘게 관대한 실행에 걸린 것).

**해결**
- `meetings.py` llm_call에 **`seed` 고정**(`_SCORING_SEED=7`) 추가 — temperature=0과 함께 쓰면
  best-effort 재현성이 확보돼 경계 문서 판정 흔들림이 크게 줄어든다.

**검증**
- 같은 문서 3회 재분석 → **54.5 / 56 / 55.5 (±1.5)**. 이전 ±32.5에서 거의 결정론으로 안정.

**교훈**
- "채점이 후하다/이상하다"는 신고를 받으면 **먼저 같은 입력을 여러 번 돌려 재현성부터 확인**할 것.
  흔들림(variance)과 편향(bias)은 원인·처방이 다르다. temperature=0 만으로 결정론을 가정하지 말 것
  (OpenAI seed 병행 필요, 그래도 완벽하진 않음 — system_fingerprint 변동).

## 3. 근거가 전혀 없는 나쁜 문서인데 데이터 점수가 너무 후하다

**증상**
- "다양한 공공데이터를 활용할 예정" 수준으로 **데이터셋 이름을 하나도 안 쓴** 문서인데
  data_utilization이 12~15/20(60~75%, acceptable~needs_improvement)로 나옴. 담당자 판단으로는
  ~5/20(25%)이 맞음.

**원인**
- 기존 [엄정 채점] 규칙은 "adequate 이상 금지" 정도로만 강제라, LLM이 **needs_improvement 상단
  (60~64%)** 에 몰아넣고 그 아래(critical_risk)로는 잘 안 내려감(주제 관련성 있으면 partial credit).

**해결**
- `reviewer_prompt.txt` [엄정 채점]을 **2단계 강제**로 재작성:
  - (하한) 구체 출처(데이터셋·기관명)·방법·수치가 사실상 전무 → **critical_risk, ≤40%**
    (2개 이상 해당·근거 전무면 ≤25%). 특히 데이터 활용은 데이터셋을 이름으로 명시 안 하면 critical_risk.
  - (중간) 방향·요소는 있으나 구체성·정량 근거가 부분적 → **needs_improvement, ≤64%**.
  - "주제·방향은 맞다"는 이유로 올리지 못하게 못 박음.

**검증 (변별력 안전점검 — 좋은 문서 안 깨졌는지까지)**
| 문서 | 총점 | 데이터활용성 |
|---|---|---|
| 나쁜 문서(출처 없음) | 55.5 | 11/20 (55%, needs_improvement) |
| 좋은 문서(A유형, 출처 풍부) | 81 | 18/20 (90%) |
- 강화 규칙은 **출처를 이름으로 명시 안 한 문서만** 잡고, A유형(HRD-Net·워크넷·고용보험 등 명시)은
  안 깎임. 55% vs 90%, 총점 55.5 vs 81로 확실히 변별.

**의사결정 / 한계**
- data를 5/20까지 강제로 내리는 **결정론적 페널티(점수 엔진에서 후처리)는 채택하지 않음** — 데이터를
  정상적으로 폭넓게 언급한 좋은 문서까지 깎을 위험이 크고, 이미 55% vs 90%로 변별되기 때문.
- LLM-판정의 **바닥 효과**(주제 관련만 있어도 ~50% 밑으로 잘 안 내려감)는 감수. 더 낮춰야 하면
  프롬프트가 아니라 결정론적 규칙이 필요한데, "출처 명시 여부"를 코드로 판별하는 게 fragile해 보류.

**교훈**
- 채점을 엄격하게 바꿀 땐 **반드시 좋은 문서로 역검증**할 것("나쁜 문서 낮추기"가 "좋은 문서 깎기"로
  번지면 변별력이 죽는다). 나쁜/좋은 한 쌍을 같이 돌려 격차가 유지되는지 본다.

## 4. A유형 ≈ B유형 (총점이 비슷) — 버그인가?

**증상**
- 작성자가 보기엔 B가 A보다 나은데 위원 총점이 A유형 ≈ B유형으로 비슷하게 나옴.

**원인 (버그 아님 — 정상)**
- 확인 결과 A·B는 **겹치는 본문이 토시 하나 안 틀리고 동일**. 차이는 내용이 아니라 **페이지 여백/
  채움 정도**(A는 여백 많음, B는 꽉 참). 즉 위원 채점(=내용)에서 A≈B는 **정상**.
- "B가 낫다"의 근거는 **분량·밀도**(문서 피드백) 축이지 위원 채점 축이 아니었다.

**해결(오히려 되돌림 + 다른 축 노출)**
- 앞서 위원 프롬프트에 넣었던 "[변별 우선] A·B를 벌려라"는 **잘못된 전제라 삭제**.
- 대신 A/B를 실제로 가르는 **분량·밀도(getFormatCheck)를 완성 리포트 "AI 피드백" 탭에 추가**
  (탭 상단 요약 카드, 점수 미반영). 같은 내용이라도 여백 많은 문서(밀도 부족)와 꽉 찬 문서(밀도
  양호)가 여기서 갈린다.

**교훈**
- "점수가 비슷하다 = 버그"가 아니다. **평가 축을 먼저 확인**할 것. 내용이 같으면 내용 점수가 같은 게
  맞고, 다른 차이(분량·밀도·서식)는 다른 축(AI 피드백)에서 드러내야 한다. 축을 섞어 프롬프트로
  억지 변별을 만들면 채점 신뢰도만 깨진다.

## 5. 분량·밀도(AI 피드백) 값이 DOCX에서 전부 None

**증상**
- `GET /workbench/{id}/format-check` 가 DOCX 대상에서 `required_pages / actual_pages /
  overall_coverage` 등 **전부 None**.

**원인**
- 분량·밀도는 문서를 **PDF로 변환해 페이지·채움률을 계산**하는데, DOCX→PDF 변환에 LibreOffice가
  필요하다. 이 환경의 LibreOffice(soffice)가 다운(`HWP_CONVERTER_UNAVAILABLE`, headless self-test
  실패)이라 DOCX는 변환 실패 → None.

**해결 / 우회**
- **PDF로 업로드하면 변환 없이 PyMuPDF로 직접 분석**되어 정상 작동(코드 변경 불필요).
  실측: SchoolBridge(꽉 찬 문서) 67% 양호 vs QA 샘플(여백 많음) 40% 부족으로 변별됨.
- 여백/빈 공간은 **레이아웃 속성**이라 애초에 PDF가 더 정확한 입력. DOCX 텍스트 추출로는 여백을 볼 수
  없다(위원 채점은 텍스트 추출이라 DOCX여도 무관).

**교훈**
- 분량·밀도 데모/테스트는 **PDF로 올릴 것**. LibreOffice 의존이 있는 기능은 이 환경에서 DOCX가 막힌다.

## 6. (환경) 갑자기 모든 API가 500 — Mongo 연결 거부

**증상**
- `/comparison`, `/format-check` 등이 갑자기 500. 로그에 `localhost:27017 [WinError 10061]
  actively refused`.

**원인**
- Mongo는 원격이고 **SSH 터널(`localhost:27017`)** 을 통해 붙는데, 터널이 끊기면 연결 거부.
  (백엔드 재시작·PC on/off 등으로 터널이 내려간 것 — 코드 문제 아님)

**해결**
- 터널 재연결: `ssh -N -L 27017:localhost:27017 kyh@220.81.7.136`
- 포트 확인: `netstat -ano | grep 27017` 에 LISTENING이 있어야 정상.

**교훈**
- DB 관련 500이 갑자기 뜨면 **코드보다 터널부터 확인**. 백엔드는 파이썬 모듈 변경 시 재시작 필요
  (`--reload` 없이 실행 중이면), reviewer_prompt.txt 같은 **프롬프트 파일은 매 호출 새로 읽어 재시작
  불필요**.

## 부록 — 확인 순서 체크리스트 (채점 이상 신고 시)

1. **재현성 먼저**: 같은 입력 2~3회 돌려 총점이 흔들리는지(variance) → 흔들리면 seed/온도 문제.
2. **어느 축인가**: 위원 채점(내용) vs AI 피드백(분량·밀도·오탈자·맥락). 축을 섞지 말 것.
3. **좋은 문서 역검증**: 엄격화가 좋은 문서까지 깎지 않는지 나쁜/좋은 쌍으로 확인.
4. **rubric 확인**: 동적 rubric이 프로젝트에 캐시(`project.dynamic_rubric_mapping`)되는지 — 프로젝트마다
   재추출되면 criterion_id·배점이 달라져 비교가 어긋난다.
5. **환경**: SSH 터널(27017) / 백엔드 재시작 여부 / LibreOffice(DOCX 변환) 상태.
