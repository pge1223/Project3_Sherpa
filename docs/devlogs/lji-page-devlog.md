# AI 피드백 워크벤치 — 만든 과정 기록

`lji-devlog.md`는 짧게 담백하게 남기는 곳이라, "AI 피드백" 워크벤치 화면 하나를
어떻게 만들었는지 그 과정만 따로 자세히 적어두는 문서. 시행착오 위주로 씀 —
결과만 보면 "그냥 PDF 띄우고 하이라이트한 거네" 싶지만, 실제로는 방향을 여러 번
갈아엎었다.

## 2026-07-21

### 1. 원문 보여주기 — 세 번 갈아엎음

**1차: 그냥 텍스트.** 파싱한 문단을 순서대로 쭉 이어붙여서 보여주는 방식으로
시작. 근데 이러면 원본 문서에 있던 굵게/기울임/제목 서식이 다 사라져서 "이게
기획서야 메모장이야" 소리 나옴.

**2차: HTML 재구성.** `docx_parser.py`에서 문단마다 run(서식 조각) 단위로
bold/italic을 뽑아내고, `html_render.py`를 새로 만들어서 `<h2>`/`<p>`/`<b>`
같은 태그로 재구성. 서식은 살아났는데, 이번엔 "이게 진짜 워드에서 보던
페이지랑 똑같이 생겨야 한다"는 요구가 나옴. 문제는 docx 파일 자체엔 "몇
페이지째 어디서 줄이 바뀌는지" 정보가 아예 없다 — 그건 워드가 화면에 그릴 때
그때그때 계산하는 값이라, HTML로는 절대 워드랑 똑같은 페이지 모양을 못
만든다는 걸 깨달음.

**3차(최종): PDF 변환.** 원본을 LibreOffice headless로 실제 PDF로 변환하고,
그 PDF를 `pdf.js`로 브라우저에 그대로 그리는 방식으로 전환. 이러면 워드/한글에서
인쇄했을 때랑 페이지 단위까지 완전히 똑같이 보인다. `preview_pdf_converter.py`를
새로 만듦 — 기존 `hwp_pdf_converter.py`(용준님, HWP 전용·시그니처 검증 있음)는
그대로 두고 완전히 별개 파일로 분리했다(색인용 변환과 화면표시용 변환은 목적이
달라서 섞으면 안 됨).

이 과정에서 실제로 겪은 버그:
- **LibreOffice 동시성 충돌**: headless 모드가 프로필(작업공간)을 하나만 써서,
  동시에 두 변환 요청이 들어오면(React 18 StrictMode가 effect를 두 번 실행해서
  실제로 재현됨) 나중 요청이 실패함. `-env:UserInstallation`으로 요청마다 독립된
  임시 프로필을 줘서 해결.

### 2. 하이라이트 매칭 — 제일 오래 걸린 부분

위원이 인용한 문단(청크)을 PDF 화면 위에서 정확히 찾아 색칠하는 기능. 원문
그대로 문자열 비교하면 될 줄 알았는데, 실제로는 4~5단계를 거쳐야 했다.

1. **원본 그대로 비교** → 실패. PDF는 화면에서 줄이 바뀌는 자리마다 실제
   줄바꿈이 생기는데, 원본 청크 텍스트는 줄바꿈 없는 문단 하나라 안 맞음.
2. **공백류를 한 칸으로 통일해서 비교** → 여전히 실패. PDF는 한글↔숫자처럼
   폰트가 바뀌는 지점마다 글자 조각이 갈라지면서, 원문엔 없던 공백이 생기기도
   하고("5년간"이 "5 년간"으로) 반대로 워드 자동 번호매기기 목록의 글머리
   기호(사설 영역 유니코드 문자로 렌더링됨)처럼 원문엔 없는 글자가 껴 있기도
   해서, "공백만 손보는" 정규화로는 부족했다.
3. **공백·기호 다 무시하고 글자/숫자만 비교** → 대부분 해결. 근데 청크 하나가
   최대 800자라 페이지 경계를 넘는 경우(실측: 예산 항목이 3페이지 끝에서
   4페이지로 이어짐)가 있어서, 페이지별로 따로 찾으면 그런 청크는 어느
   페이지에서도 못 찾았다.
4. **페이지별이 아니라 문서 전체를 하나로 이어붙여서 한 번에 검색** → 최종
   해결. 문서 전체를 한 좌표계로 보고 찾은 뒤, 찾은 범위가 어느 페이지의 어느
   조각들에 걸치는지 역산해서 하이라이트를 그린다.

겹치는 하이라이트(같은 문단을 여러 피드백이 가리키는 경우) 처리도 이 과정에서
같이 정리함 — 원래는 "먼저 온 것만 남기고 나머지 버림"이었는데, 이러면 뒤에
겹친 피드백이 화면에서 그냥 사라져서, 완전히 같은 위치를 가리키는 것끼리는
하나로 묶어 보여주는 방식으로 바꿈.

### 3. "선택됨" 표시 디자인 — 아직 미완성

하이라이트를 클릭했을 때 "이게 지금 선택된 거다"를 보여주는 방식을 여러 번
바꿨다: 조각마다 테두리 → 밑줄 느낌의 아래쪽 선 → 선택 영역 전체를 감싸는
통짜 박스(스코프 버그로 엉뚱한 문단까지 뒤덮은 적 있음, 고침) → 줄 단위로
나눠서 박스 여러 개. 근데 마지막 방식도 "맘에 안 든다"는 피드백을 받았고,
디자인 자체는 나중에 다시 다루기로 하고 일단 넘어감(별도 메모 남겨둠).

### 4. 맥락 이상 감지 — 통계에서 LLM으로 갈아탐

강사님이 요청한 "이 내용이 여기 왜 있는지 모르겠다" 기능. 처음엔 "AI 호출 없이
공짜로 하자"는 생각으로 접근했다가 결국 못 버티고 LLM 방식으로 갈아탐.

1. **RAG 청크 임베딩 통계로 이상치 찾기** → 청크 하나가 여러 문단을 묶어버려서
   (chunk_size=800) 끼워 넣은 이상 문단이 평균에 묻혀 아예 안 잡힘.
2. **문단 단위로 다시 임베딩해서 통계** → 완전히 다른 주제(반려동물, 암호화폐
   얘기를 일부러 끼워 넣어 테스트)는 잡았는데, "같은 주제인데 타겟층만 슬쩍
   다른" 미묘한 경우(청년 대상 서비스에 노년층 얘기가 끼어드는 것)는 임베딩
   거리 자체가 크게 안 벌어져서 못 잡음. 임베딩 방식 자체가 이런 미묘한
   불일치를 잡을 수 있는 신호가 아니었다.
3. **GPT가 전체 문서를 한 번에 읽고 후보를 찾은 뒤, 다시 깐깐하게
   재검증하는 2단계 방식** → 최종. 미묘한 경우까지 다 잡았다. 대신 결과가
   실행마다 흔들려서(같은 문서·같은 프롬프트인데 후보가 나왔다 안 나왔다 함)
   `temperature=0`으로 고정해서 안정화.

테스트용으로 실제 공모전 공고문(사토리 사이트에서 확인)에 맞춘 10페이지짜리
기획서를 직접 써서, 안 어울리는 문단 2개를 일부러 끼워 넣어 검증했다.

### 5. "왜 이 항목 피드백이 없지?" — 위원 평가 누락 조사

사용자가 "AI 피드백이 부족해 보인다"고 해서 몽고DB로 실제 회의 기록을 파본
결과, 위원이 rubric 항목을 채점 못 하는 경우가 3가지였다:

1. 위원이 직접 "근거 부족"이라고 판단
2. 위원은 판단(예: "적당함")까지 냈는데, RAG 근거 연결 시스템이 그 판단을
   뒷받침할 근거를 못 찾아서 최종 반영이 막힘
3. GPT가 그 항목 자체를 응답에서 통째로 빼먹음(지시 불이행)

지금까지는 세 경우 다 `transform.py`가 조용히 버려서, 사용자는 그 항목이 왜
없는지 알 방법이 없었다. `review_output.schema.json`을 v2.3.0으로 올려서
`unscored_criteria` 필드를 추가하고, 1·2번은 화면에 이유와 함께 보여주고
(2번은 "위원이 실제로 낸 판단"까지 같이 보여줘서 "당신 탓 아니다"를 명확히
함), 3번은 사용자한테 안 보여주는 대신 그 자리에서 GPT한테 재질문해서 즉시
채우는 로직을 추가했다.

중간에 실측으로 또 발견한 문제: rubric 전체가 위원 프롬프트에 그대로 노출되다
보니, 위원이 자기 담당 아닌 기준까지 "어차피 보이니까" 의견을 내는 경우가
있었고, 그건 RAG가 애초에 그 (위원,기준) 조합으로 근거를 준비한 적이 없어서
항상 근거연결 실패로 뜬다 — 이게 "평가되지 않은 항목"을 잔뜩 부풀리는 노이즈여서,
위원이 실제로 맡은 범위 밖의 항목은 화면에 아예 안 보여주게 필터링 추가.

  못 여는 근본적인 한계가 있다는 걸 실측으로 확인 — 우리 코드 버그가 아니라
  LibreOffice의 알려진 한계. 다음에 팀이랑 어떻게 할지 논의 예정.
