## 1. 프로젝트명

AI Review Board

## 2. 한 줄 정의

업로드된 문서와 평가기준을 바탕으로 여러 AI 전문가가 독립 검토하고, 근거와 규칙에 따라 최종 평가 리포트를 만드는 RAG 기반 AI 심사위원회 시스템.

## 3. 해결하려는 문제

사업계획서, 제안서, 공모전 제출문서는 여러 평가축을 동시에 충족해야 한다. 사용자는 무엇이 부족한지, 어떤 문단을 고쳐야 하는지, 실제 평가기준과 연결되는지 파악하기 어렵다. 일반 LLM 피드백은 근거가 불분명하거나 점수가 일관되지 않을 수 있다.

AI Review Board는 다음을 제공한다.

- 여러 전문가 관점의 분리된 검토
- 문서 근거와 평가기준을 연결한 설명
- 규칙 기반의 일관된 점수
- 우선 수정 항목과 구체적 개선안
- AI 위원 회의의 시각화

## 4. 주요 사용자

- 사업계획서나 지원서 작성자
- 스타트업 및 예비창업자
- 공모전·지원사업 참가자
- 기획서와 제안서를 검토하는 팀
- 문서 품질을 반복 개선하려는 사용자

## 5. 핵심 사용자 흐름

1. 로그인한다.
2. 프로젝트를 생성한다.
3. 평가 대상 문서를 업로드한다.
4. 평가기준 문서 또는 공고문을 업로드한다.
5. 시스템이 문서를 파싱·청킹·임베딩한다.
6. 시스템이 도메인을 분류한다.
7. 사용자에게 분류 결과를 보여주고 필요하면 수정하게 한다.
8. 위원별 RAG 검색을 수행한다.
9. 각 AI 위원이 독립 평가한다.
10. Python 점수 엔진이 평가기준 충족도를 계산한다.
11. 위원장이 합의·이견·수정 우선순위를 종합한다.
12. 결과를 저장하고 웹 리포트로 보여준다.
13. 핵심 위원 2명의 발언을 TTS·립싱크 영상으로 재생한다.
14. 수정 문서를 다시 평가해 전후 결과를 비교한다.

## 6. 핵심 가치

- 근거성: 모든 핵심 지적은 원문 출처와 연결한다.
- 다중 관점: 위원별 역할과 평가축을 분리한다.
- 일관성: 최종 점수는 규칙 엔진이 계산한다.
- 추적 가능성: 문서명, 페이지, 섹션, 청크 ID를 유지한다.
- 개선 지향: 단순 판정이 아니라 수정 이유와 우선순위를 제공한다.
- 시각화: AI 회의를 2인 화상회의 형태로 표현한다.

## 7. 성공 기준

- 핵심 피드백에 유효한 근거가 연결된다.
- 동일 입력과 동일 규칙에서 점수가 재현된다.
- 위원별 평가가 서로 다른 역할 관점을 유지한다.
- 근거 부족 시 추정 대신 부족 상태를 표시한다.
- 사용자가 결과를 보고 수정할 위치와 이유를 이해할 수 있다.
- 일부 영상 생성 실패가 전체 결과 조회를 막지 않는다.

## 기술 의사결정 원칙

기술 선택은 해당 영역 담당자가 주도하지만 단독 확정하지 않는다. 다른 모듈의 인터페이스, 데이터 구조, 배포·운영 방식 또는 담당 범위에 영향이 발생하면 관련 담당자와 영향 범위를 협의하고 팀 동의를 받은 뒤 확정한다.

현재 Vector DB는 미확정이며, RAG 담당자가 후보를 비교·제안한다. 최종 선정 전에는 LangGraph가 소비하는 Retriever JSON 계약과 NCP 배포·운영 영향을 각각 관련 담당자와 검토한다.
