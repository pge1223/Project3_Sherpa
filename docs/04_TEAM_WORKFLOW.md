# 04. Team Workflow

## 1. 최신 역할 분담

### 가은 — PM · React 프론트 · Persona/Prompt 기획

- 전체 일정과 요구사항 관리
- 사용자 흐름과 화면 구조
- React 프론트 개발
- Persona Card 기획
- 위원별 평가 관점과 말투 정의
- 회의 순서와 회의 문화 정의
- Reviewer Prompt, Chair Prompt 초안
- Mock API와 샘플 JSON 기반 선개발

담당 경계:

- LangGraph 실행 코드는 경이 담당
- RAG 검색 구현은 용준 담당
- DB와 API는 윤한 담당
- 영상 생성은 재인 담당

### 윤한 — FastAPI · MongoDB · CRUD · NCP

- FastAPI 통합 허브
- MongoDB 모델과 CRUD
- 사용자·프로젝트·문서·회의·결과 데이터 관리
- 파일과 미디어 저장
- 비동기 Job과 상태 API
- 인증·권한·보안·로그
- OpenAPI
- NCP 배포

### 경이 — LangGraph · 점수 엔진 · 평가 결과 구조

- LangGraph State, 노드, 엣지
- 위원별 독립 평가
- 위원장 종합
- Python 규칙 기반 점수 계산
- 누락 감점과 가중치
- 수정 우선순위
- 평가 결과 JSON 구조
- media_script 생성
- 회의 품질과 일관성 테스트

### 용준 — RAG 파이프라인

- PDF, DOCX, PPTX 파싱
- 페이지·슬라이드·문단 메타데이터
- 청킹과 오버랩 실험
- KURE-v1 임베딩
- Vector DB 후보 비교, PoC, 관련 담당자 협의 및 최종안 제안
- 선정 제품의 컬렉션·인덱스 구조 설계
- 역할별 검색과 Top-K
- 근거 부족 판정
- 출처 연결
- 도메인 분류
- Golden Set과 RAG 평가

### 재인 — AI Human 영상

- 핵심 페르소나 2명의 외형·복장·음성
- TTS
- MuseTalk 립싱크
- 발언 단위 영상
- 재생 큐와 발언 순서
- 영상 생성 상태
- 이미지 + TTS 폴백
- 2분할 회의 화면 연동

## 2. 병렬 개발 원칙

```text
공통 JSON 계약 확정
→ Mock 데이터 작성
→ 모듈별 병렬 개발
→ 부분 통합
→ 전체 통합
```

### 병렬 작업

- 가은: Mock API 기반 React와 Prompt Spec
- 용준: 샘플 문서 기반 RAG
- 경이: Mock Retriever 기반 LangGraph와 점수 엔진
- 윤한: Mock 평가 결과 기반 API, DB, Job
- 재인: 더미 발언 기반 TTS와 영상

## 3. 부분 통합 순서

```text
용준 RAG ↔ 경이 LangGraph·점수
경이 LangGraph ↔ 윤한 FastAPI·MongoDB
윤한 API ↔ 가은 React
경이 media_script ↔ 재인 AI 영상
재인 영상 ↔ 윤한 미디어 저장·상태 API
윤한 미디어 API ↔ 가은 영상 재생 UI
```


## 4. 기술 선택 및 협의 원칙

- 각 기술 영역 담당자가 후보 비교와 최종 제안을 주도한다.
- 다른 담당자의 인터페이스, 데이터 구조, 배포 환경 또는 작업 범위에 영향이 있으면 단독으로 확정하지 않는다.
- 주관 담당자는 영향받는 담당자에게 후보, 선정 기준, 예상 영향 범위를 공유한다.
- 관련 담당자는 자신의 담당 영역에 대한 검토 의견을 제공한다.
- 팀 동의 후 최종 기술을 확정하고 관련 문서에 반영한다.

### Vector DB 협의 예시

| 구분 | 담당 | 검토 내용 |
|---|---|---|
| 주관 | 용준 | 후보 비교, 검색 성능, 메타데이터 필터, PoC, 최종안 제안 |
| 협의 | 경이 | Retriever JSON, LangGraph 소비 구조, Schema 영향 |
| 협의 | 윤한 | NCP 배포, 영속성, 백업, 모니터링, 비용 |
| 조건부 협의 | 가은 | 검색 상태, 출처 표시, 오류 응답 등 프론트 계약 영향 |

## 5. 공통 계약 변경 규칙

```text
변경 제안
→ 영향 담당자 확인
→ JSON 예시 공유
→ 팀 동의
→ schema_version 갱신
→ Mock 갱신
→ 코드와 테스트 반영
```

- 기존 필드 삭제보다 선택 필드 추가를 우선한다.
- 공통 계약은 담당자 개인 브랜치에서 임의 확정하지 않는다.
- `contracts/` 변경은 관련 담당자 리뷰를 받는다.

## 6. Git 작업 규칙

- 기능 단위 브랜치를 사용한다.
- PR에는 변경 목적, 영향 모듈, 테스트 방법을 적는다.
- 공통 Schema 변경 PR에는 변경 전후 JSON 예시를 포함한다.
- 대용량 원본 문서와 생성 영상은 Git에 직접 올리지 않는다.
- `.env`, API Key, 개인 인증정보를 커밋하지 않는다.

## 7. 완료 기준

- 모듈이 Mock 입력으로 실행된다.
- JSON Schema 검증을 통과한다.
- 실패 상태와 오류 메시지가 정의되어 있다.
- 최소 테스트가 존재한다.
- README 또는 담당 모듈 문서가 갱신되어 있다.
- 다른 담당자가 계약만 보고 연결할 수 있다.
