# 채점 기준 명세 — Codex 핸드오프 (완성 리포트 점수 엔진)

> 작성: 경이(mky) / 2026-07-24
> 목적: LLM 채점의 "판단"을 **코드·테스트로 고정**하기 위한 기준 명세. Codex가 이걸 근거로
> (1) reviewer 프롬프트 보정 (2) 결정론적 점수 상한 로직 (3) 회귀 테스트를 구현할 수 있게 함.
> 이 문서의 모든 점수는 "근거를 든 채점"이며, 각 항목마다 왜 그 점수인지(rubric 대비)를 명시한다.
>
> 관련 파일: `ai/meeting/prompts/reviewer_prompt.txt`, `backend/app/api/routes/meetings.py`(llm_call),
> `ai/meeting/scoring/calculator.py`, `ai/meeting/scoring/comparison.py`

## 0. 배경 (이미 한 것 / 남은 것)

- **문제**: 같은 문서를 채점해도 총점이 31.5 ↔ 64로 흔들리고(±32.5), 근거 없는 문서에도 후한 점수.
- **이미 한 것 (Claude, PR #151)**:
  - `seed` 고정 + `temperature=0` → 같은 문서 3회 54.5/56/55.5(±1.5)로 흔들림 해소.
  - reviewer 프롬프트 [엄정 채점] 2단계 강제(하한 critical_risk / 중간 needs_improvement).
- **한계 (여기서 Codex에게 넘김)**: 프롬프트만으로는 LLM이 "주제 관련 있으면 ~50% 밑으로 잘 안 감"
  (바닥 효과). **근거 없는 문서를 확실히 30점대 이하로 내리려면 코드에서 결정론적 상한을 적용**해야 함.

## 1. 평가기준표 (rubric) — 이 공모전 동적 rubric

| criterion_id | 항목명 | 배점 | 측정 대상(무엇을 보는가) |
|---|---|---:|---|
| `ai_innovation` | AI 혁신성 | 25 | AI를 **어떤 방법(모델·알고리즘·절차)** 으로 쓰는가, 규칙기반 대비 차별성 |
| `data_utilization` | 데이터 활용성 | 20 | **어떤 공공데이터(데이터셋·기관명)** 를 **어떤 필드**로, 어떻게 결합·가공하는가 |
| `feasibility` | 실현 가능성 | 20 | 구현 방법·기술 스택·단계·자원·리스크가 **구체적**인가 |
| `creativity_differentiation` | 창의성·차별성 | 15 | 기존 서비스 대비 **구체적으로** 무엇이 다른가 |
| `expected_effect` | 기대 효과성 | 20 | 효과가 **정량·근거**와 함께 제시되는가(As-Is/To-Be) |
| | **합계** | **100** | |

> ⚠️ **rubric은 공고문에서 동적 추출**되어 프로젝트마다 criterion_id·배점이 달라질 수 있고
> `project.dynamic_rubric_mapping`에 캐시된다. **회귀 테스트는 이 rubric을 고정 fixture로 박아
> 돌릴 것**(동적 추출까지 물리면 변수가 늘어 재현이 깨진다).

## 2. 판정 밴드 (judgment → 배점 대비 %)

| judgment | 배점 대비 % | 의미 |
|---|---|---|
| `critical_risk` | **0 ~ 40%** (근거 전무면 0~25%) | 핵심 요건 미충족, 근거·출처·수치·방법 전무, 추상적 서술뿐 |
| `needs_improvement` | **40 ~ 64%** | 방향·요소는 있으나 구체성·정량 근거·실행 계획이 부분적 |
| `adequate` | **65 ~ 84%** | 요건 충족 + 문서 근거 있음, 일부 보완 여지 |
| `strong` | **85 ~ 100%** | 정량 근거·구체 실행·차별성까지 충분히 뒷받침 |

## 3. 점수 상한 규칙 (결정론적 보정에 바로 쓸 형태)

각 항목마다 아래 **신호를 코드로 카운트**해 상한을 적용한다. 상한은 "그 이상 주지 않는다"는 **캡**이지
하한이 아니다(더 낮게는 허용). LLM 점수가 상한을 넘으면 상한으로 clamp.

| 신호 | 판별(결정론적) | 적용 항목 | 상한 |
|---|---|---|---|
| **S1. 출처 전무** | 알려진 데이터셋/기관명 사전(HRD-Net, 워크넷, 고용보험, 국가기술자격, 내일배움카드, 공공데이터포털 등)과 **0건 매칭** | `data_utilization` | **배점의 25%** (critical_risk) |
| **S2. 수치 전무** | 문서(또는 해당 섹션)에 숫자·정량 표현(%, 배, 명, 원, 건, 개월 등) **0개** | 모든 항목 | 배점의 **40%** |
| **S3. 미실현 서술 위주** | "예정/계획/할 것이다/가능할 것이다/기대된다" 류 어미 비율이 문장의 **50% 초과** | 해당 항목 | 배점의 **40%** |
| **S4. 방법 전무** | 구체 방법 키워드(임베딩/모델/알고리즘/파이프라인/API/단계/정규화 등) **0개** | `ai_innovation`, `feasibility` | 배점의 **40%** |
| **S5. 공고 필수요건 미언급** | 공고가 요구한 필수 요소(예: 특정 데이터·기능) 미포함 | 해당 항목 | 배점의 **50%** |

- **복수 신호 동시 → 가장 낮은 상한 우선**. (예: data에서 S1+S2+S3 동시 → 25%가 아니라 더 낮게,
  **배점의 15% 이하**로 clamp 권장.)
- 이 규칙은 "**근거 없음/추상/요건 미충족 시 점수 상한**"에 정확히 대응한다(Codex 요청 4번).

## 4. 문서별 적정 점수 (회귀 테스트 기준값) — 내 판단

> 각 문서의 **적정 총점·항목별 점수**와 그 rubric 근거. Codex는 이 값을 회귀 테스트의 기대값으로 삼되,
> LLM 특성상 ±3점 허용오차(항목당 ±20% band 이내)로 검증할 것. **밴드(판정)가 맞는지**가 핵심.

## 7. 주의 / 함정 (Codex가 알아야 할 것)

1. **rubric은 동적 추출·프로젝트별 캐시**(`project.dynamic_rubric_mapping`). 회귀 테스트는 rubric을
   고정 fixture로 주입해 돌릴 것. 안 그러면 criterion_id/배점이 달라져 기대값이 어긋난다.
2. **채점 흔들림은 seed로 줄였지만 완전 결정론은 아니다**(OpenAI system_fingerprint 변동). LLM 점수를
   그대로 assert하지 말고, **§3 결정론 상한을 적용한 뒤**의 값을 assert하면 안정적이다.
3. **좋은 문서 역검증 필수**: 상한 규칙이 A유형(출처 명시) 같은 좋은 문서를 깎으면 안 된다. S1은
   "출처 0건"에서만 발동 — A유형은 출처 5개라 미발동. 테스트 세트에 반드시 우수 문서를 포함해 회귀 방지.
4. **점수는 위원 제안(score_recommendation)이 아니라 점수 엔진(criterion_owner·결정론) 산출이 최종**.
   상한 보정은 이 최종 계산 단계(`calculator.py`)에 거는 게 자연스럽다.

## 8. 실측 요약 (참고 · 현재 엔진 상태)

- 흔들림 해소 후(seed): 50점대테스트 문서 3회 **54.5 / 56 / 55.5** (현재 엔진 = 아직 상한 보정 전).
- **목표**: §3 상한 보정을 얹으면 이 문서가 **~27점(§4-1)** 으로 내려가야 한다(data 5/20).
- 우수 문서 A유형: **81점, data 18/20(90%)** — 상한 보정 후에도 유지돼야 한다(변별력).
